## 2. 执行模式

### 2.1 调试模式

若输入不是以用例 Excel 启动的正式测试任务（即用户通过自然语言发起的浏览、抓取数据、点击操作等），视为**调试模式**。

- **无须输出测试报告**
- 按需操作即可，灵活执行

---

### 2.2 输出要求

正式用例测试完成后**必须输出结构化报告**，并保存到 `<时间戳>/test_report.md`。

模板和规则见 `test-report` 子技能。

---

### 2.3 执行策略

> **重要提醒：任务可能非常多且复杂，涉及大量用例的连续执行。以下规则必须严格遵守。**

#### 2.3.1 全局连续执行

- 用例之间默认连续执行，绝不主动暂停
- 若用户未明确指示暂停，整个测试流程必须一气呵成执行到底
- 多条用例必须**串行执行**，禁止并行

#### 2.3.2 单用例内连续执行

- 单用例内多步骤之间不得无故暂停
- 必须连续执行到该用例完成，禁止在过程中插入无关操作或无故中断

#### 2.3.3 禁止在单用例中询问用户

- 严禁在执行过程中向用户提问、请求确认或输出状态等待回复
- 每一步操作完成后直接进入下一步，不需要用户确认
- 只有在整个用例完全执行完毕后，才输出测试结果和报告

#### 2.3.4 禁止自主关闭页签

- 严禁调用 `close_page` 关闭任何页签
- `close_page` 仅在用户明确要求时才调用
- 需要切换页签时用 `select_page`

**Why:** 页签关闭后不可恢复，可能丢失关键页面状态和上下文信息。

#### 2.3.5 异常处理

- 只有不可恢复的异常才退出当前用例（服务器 500、页面完全无法加载、网络不可达、用例信息完全缺失），并在报告中说明原因
- 可恢复异常（会话超时、角色切换等）不属于此范畴，遵循 2.3.6/2.3.7 处理
- 退出当前用例后，继续执行后续用例，不得影响其他用例

#### 2.3.6 会话超时处理

- 审批系统等多步骤流程中，**会话超时不是中止理由**。遇到登录超时、强制退出等情况时，**必须重新登录并继续完成当前用例的全流程**
- 重新登录后如果之前填写的表单数据丢失，**必须重新填写所有字段**，不得跳过任何步骤
- **禁止**因会话超时、重新登录后需要大量时间就提前生成测试报告或标记用例为 PARTIAL/FAIL
- 只有系统真正异常（如服务器 500、页面完全无法加载、网络不可达等）才可标记为 FAIL
- 重新登录后需从当前步骤**回退到该步骤起点**，重新执行该步骤及后续所有步骤
- 每个用例必须**完整执行到最终预期结果验证**，不得半途而废

**Why:** 会话超时是测试环境常见问题，不能成为中止测试的理由。用户需要完整的测试覆盖，不完整的报告没有实际意义。

**How to apply:** 每次遇到会话超时：1) 记录超时状态；2) 重新登录系统；3) 导航回当前步骤起点；4) 完整重新执行该步骤及后续所有步骤；5) 只有用例完全完成后才输出报告。

#### 2.3.7 角色/用户切换处理

- 用例步骤中涉及切换用户或角色时，**禁止因此中断当前用例的执行**。
- 必须完成当前用户/角色的操作步骤后，再执行切换操作，继续完成同一用例的剩余步骤。
- 切换用户时：1) 按用例要求退出当前用户或清空会话；2) 用新用户的凭据重新登录；3) 回到切换步骤的起点，重新执行该步骤及后续所有步骤；4) 每个用例必须完整执行到最终预期结果验证，不得半途而废。
- **禁止**因需要切换用户就提前终止用例，或跳过切换后的步骤。
- 切换用户后如果之前填写的表单数据丢失，**必须重新填写所有字段**，不得跳过。
- 只有系统真正异常（如服务器 500、页面完全无法加载、网络不可达等）才可标记为 FAIL。

**Why:** 多角色/多用户切换是测试用例的常见场景，不能因为需要切换用户就中止测试。用户需要完整的测试覆盖，不完整的报告没有实际意义。

**How to apply:** 当用例步骤要求切换用户/角色时：1) 记录当前步骤起点；2) 执行退出/切换操作；3) 用新用户凭据登录；4) 导航回切换步骤起点；5) 完整重新执行该步骤及后续所有步骤；6) 只有用例完全完成后才输出报告。

## 4. 执行方式

### 4.0 禁止派遣 Agent 代理

- **严禁使用 `Agent` 工具派遣任何子代理来执行浏览器操作或测试步骤。**

**Why:** `Agent` 子代理无法共享浏览器上下文（页签、DOM、cookie 等），派遣代理会导致浏览器状态丢失、页签切换失败、表单填写异常等执行错误。

### 4.1 任务创建

为每个用例创建独立任务（使用 TaskCreate 工具），将用例完整详情带入 `description`。

> **注意：TaskCreate 工具是允许的。4.0 禁止的是 Agent 工具（派遣子代理），两者不同。Agent 子代理无法共享浏览器上下文，会导致浏览器状态丢失。**

| 字段 | 格式 |
|------|------|
| `subject` | `TC-序号: 用例名称（简写）`，如 `TC-1: 登录测试` |
| `description` | 包含用例编号、名称、描述、预置条件、测试步骤、预期结果 |

### 4.2 用例编号与目录规则

1. 用例中明确包含编号的，**直接使用**，如 `FRS07_CSZSQ_01_002_001`
2. **禁止**自行编造或修改用例编号（添加前缀后缀、替换字符、截断、合并）
3. 没有明确编号时，使用 `TC-序号` 作为编号

#### 目录结构

同一个项目下多次执行的结果按**时间戳文件夹**区分存放，每个时间戳文件夹内包含该次执行的所有用例截图（**不按用例再分子文件夹**）。

```
当前工作目录/
├── 20260623_143000/          ← 时间戳文件夹（本次执行）
│   ├── TC-1-Step01.png
│   ├── TC-1-Step02.png
│   ├── TC-1-Step03-01.png    ← Step03第1次截图（若某步骤有多个关键操作）
│   ├── TC-1-Step03-02.png    ← Step03第2次截图（若某步骤有多个关键操作）
│   ├── TC-1-Step04.png
│   ├── TC-1-Step05.png
│   ├── TC-1-Step06.png
│   ├── TC-2-Step01.png
│   └── test_report.md
├── 20260623_150000/          ← 时间戳文件夹（下次执行）
│   └── ...
└── ...
```

**创建执行目录：** 首次执行时用 `YYYYMMDD_HHmmss` 格式创建时间戳文件夹。获取时间戳的方法：

```bash
python "${CLAUDE_SKILL_DIR}/get-timestamp/get_timestamp.py"
```

返回如 `20260623_143000`，然后用此值创建目录（如 `mkdir 20260623_143000`）。所有截图和报告统一保存在 `<时间戳>/` 目录下。

所有网页操作必须使用 **`mcp__plugin_chrome_devtools`**提供的MCP工具。

**使用规则：**
- **禁止**通过 `Bash` 执行 `npx chrome-devtools-mcp` 命令来间接调用
- **禁止**使用其他浏览器自动化工具（Selenium / Playwright / Puppeteer 等）

| `press_key` | 键盘操作（Ctrl+A, Delete, Backspace, Enter 等） | **禁止传 `includeSnapshot`** |
| `hover` | 鼠标悬停 | **禁止传 `includeSnapshot`** |
| `drag` | 拖拽元素 | **禁止传 `includeSnapshot`** |

| `handle_dialog` | 处理浏览器弹窗（alert/confirm/prompt） | — |
| `evaluate_script` | 执行 JS 代码（**绝对禁止使用**，除非用户明确要求） | — |

**工具参数核心约束：**
- **`click` / `fill` / `drag` / `press_key` / `hover` / `upload_file` 一律禁止传 `includeSnapshot` 参数**
- `fill` 是**替换操作**，会直接清空已有内容后填入新值。**禁止在 `fill` 之前使用 `Ctrl+A`、`Delete`、`press_key` 等额外清除操作**，直接 `fill` 即可。
- **调用 take_snapshot 时必须指定 filePath**: 始终传入 filePath，将快照保存到本地文件，再考虑如何处理。

### 禁止使用的工具

- **`evaluate_script` — 绝对禁止使用，除非用户明确要求。** 该工具会执行任意 JS 代码，可能干扰页面状态、触发意外行为。所有操作必须通过 `click` / `fill` / `take_snapshot` 等安全工具完成，绝不通过 JavaScript 注入操作表单或页面。
- **`wait_for`、`Bash sleep` — 绝对禁止使用。** 大模型自身交互已有延迟，该工具会在工具调用层面额外阻塞等待，导致执行效率急剧下降、超时风险大增。页面加载等待应通过 `take_snapshot` / `take_screenshot` 直接检查当前状态来替代，而非主动等待。**替代方案**：操作完成后直接调用 `take_screenshot`，通过截图/快照确认页面状态。如果页面未就绪，再次调用 `take_screenshot` 即可，无需主动等待。

## 7. 基本工作流

### 7.1 截图场景、执行流程与截图规范

> **性能提示：`take_screenshot` 和 `take_snapshot` 可以并行调用，无需等待彼此返回，以缩短整体执行时间。**

#### L1 — 关键验证点（必须截图 + Read 读图）

| 场景 | 说明 |
|------|------|
| 首次页面加载 / 登录后 | 确认登录成功（用户头像/用户名） |
| 页面导航 / 切换页签后 | 确认页面内容正确 |
| hover 悬停 | 查看鼠标悬停弹出的菜单、提示 |
| 搜索 / 筛选后 | 确认列表正确刷新 |
| 表单提交前 / 表单提交后 | 确认跳转成功页或错误提示 |
| 弹窗出现或关闭 | 确认弹窗标题、内容、表单字段 |
| 用例预期结果验证（断言点） | 确认所有断言条件满足 |
| 操作失败需要诊断时 | 截图作为问题证据 |

**L1 完整场景清单**：登录成功、菜单切换、页签切换、搜索筛选、下拉框选择（展开+选项确认）、日期选择（展开+日期确认）、单选/复选操作、文件上传、表单提交前、表单提交后、弹窗出现、成功提示、错误页面

**L1 读图验证强制规则：**
- **截图不是终点，Read 读图才是验证步骤**：`take_screenshot` 后必须用 `Read` 读取截图
- **结果判定必须基于截图证据**：禁止臆断，必须看到实际 UI 变化
- **不要假设"操作成功=预期达成"**

**每次 Read 截图后必须检查：**

| 检查项 | 说明 |
|--------|------|
| 页面概览 | 是否正确导航到预期页面 |
| 用户状态 | 是否已登录，用户名是否正确 |
| 表单字段 | 填写的值是否与预期一致 |
| 按钮状态 | 按钮文字、颜色、是否可点击 |
| 弹窗内容 | 标题、表单字段、错误/成功提示 |
| 列表数据 | 是否显示了正确的记录 |
| 错误信息 | 是否有错误提示，内容是否正确 |
| 页面布局 | 关键元素是否在正确位置 |

###### 执行流程

###### [定位]

**信息足够**
> `take_snapshot` → `Grep(-C=N)` 搜关键词 → 确认 uid

**找不到 → 兜底**
> `take_screenshot` + `Read` → 回溯定位

**信息不足**
> `take_screenshot` + `Read` → `take_snapshot` → `Grep(-C=N)` 搜关键词 → 确认 uid

###### [操作]

`click` / `fill` / `type_text` / `hover` / `upload_file` 等

###### [验证]

| 层级 | 策略 |
|------|------|
| 非 L1 | 不截图，直接回到 [定位] |
| L1 | `take_screenshot` + `take_snapshot` → 验证 → 记录 → 继续 |

##### 截图规范

| 规则 | 说明 |
|------|------|
| 保存路径 | 必须指定 `filePath` 保存到本地 |
| 命名格式 | `<用例编号>-Step<序号>[ -N].png`，如 `TC-1-Step01.png`、`TC-1-Step03-01.png` |
| 后缀编号 | 同一个步骤多次截图用 `-01`/`-02` 后缀区分 |
| 编号连续 | 每个用例的 Step 序号从 01 开始连续编号，不跳过 |
| 保存目录 | 保存在 `<时间戳>/` 目录下，不按用例再分子文件夹 |

**命名示例：**
- `TC-1-Step01.png` — TC-1用例第1步
- `TC-1-Step02.png` — TC-1用例第2步
- `TC-1-Step03-01.png` — TC-1用例第3步第1次截图
- `TC-1-Step03-02.png` — TC-1用例第3步第2次截图
- `FRS07_CSZSQ_01_002_001-Step01.png` — 使用原始编号的用例第1步

### 7.2 快照管理

| 规则 | 说明 |
|------|------|
| 保存位置 | `take_snapshot` 必须先保存至 `temp/snapshot.txt`（覆盖写入） |
| 优先默认 | **优先使用默认 take_snapshot（verbose=false），元素找不到时切换 take_snapshot(verbose=true) 获取完整元素树** |
| 读取方式 | **必须将快照保存为临时文件，再用 Grep(-C=N) 根据关键词搜索，读取快照的局部** |
| 复用策略 | 同一页面未操作时可复用已有快照，多次 Grep 搜索不同元素 |
| 刷新策略 | **每次交互操作（click/fill/submit 等）后快照会刷新，必须重新调用 `take_snapshot` 获取最新快照** |

#### 定位元素 uid 方法

核心原则：**用 `Grep(-C=N)` 一步拿到匹配行 + 相邻行，直接从中提取 uid。禁止使用 Grep 和 Read 之外的任何方式读取 snapshot 文件。**

1. **保存快照**：`take_snapshot` 固定保存为 `temp/snapshot.txt`
2. **Grep 搜索**：必须使用 `Grep` 内置工具搜索关键字，并设置 `-C=N`（如 `-C=5`）获取匹配行前后上下文，直接从中读取 uid
3. **确认上下文**：确认提取的 uid 确实对应目标元素

> `Grep(-C=N)` 一步返回匹配行 + 前后 N 行，uid 通常就在匹配行的相邻行内，`-C` 值根据元素与关键词的间距调整。
> **读取 snapshot 文件（temp/snapshot.txt）时，仅允许使用 `Grep` 和 `Read` 工具。**

### 7.3 页签管理

#### 检查新页签

> 点击登录、提交表单、打开链接等可能触发新页签的操作后，执行 `list_pages` 确认，如有新页签则用 `select_page(bringToFront=True)` 切换。

#### 系统切换

- 进入新系统：多系统切换的用例流程中，**禁止在现有页面上直接 `navigate_page` 到新系统**。必须通过 `new_page` 打开新页签进入新系统。
- 切换页签：使用 `list_pages` + `select_page(bringToFront=True)` 切换页签。

**Why:** 多系统切换是用例的常见场景，直接 navigate 会覆盖当前页面状态，导致无法返回。

## 8. 特殊 UI 组件表单处理

本章节适用于各种前端框架（Element UI、Ant Design、Vuetify 等）中的**特殊 UI 组件**。通用工具无法直接操作的场景，优先尝试通用方式，仍失败则参考本章处理。

### 8.1 通用工具优先

**绝对禁止使用 `evaluate_script`（JS 注入），除非用户明确要求。** 表单框架（Vue / React / Element UI 等）使用自定义状态管理，JS 赋值不触发组件更新，且可能干扰页面状态。对任意框架的表单，**优先使用通用工具，逐个字段填写**：

| 工具 | 适用场景 |
|------|----------|
| `fill` | 单字段填写：文本框、非 readonly 输入框 — **fill 是替换操作，会直接清空并填入新值，禁止在 fill 前使用 Ctrl+A、Delete、press_key 等清除手段。**。 |
| `click` | 单选/复选框、下拉框展开及选项选择、日期数字选择、按钮操作 |

**禁止使用 `fill_form` 批量填写。** 表单通常包含多种类型的框体（下拉框、日期选择器、复选框、只读字段等），`fill_form` 无法区分字段类型，会导致误填只读字段、跳过特殊控件、或触发错误。必须逐个字段使用 `fill` / `click` 分别处理。

### 8.2 iframe 处理

表单通常嵌套在 iframe 中。`click` / `fill` 等工具会自动跨 iframe 定位元素，无需手动处理。

### 8.3 下拉框操作

特征：许多框架的下拉框是 `readonly textbox`，**但不能用 `fill` 填写**，且下拉选项在展开前不可见。

```
步骤:
1. take_snapshot + Grep(-C=5) → 找到 readonly 的 textbox（下拉框组件）的 uid
2. click 该 textbox uid → 展开选项列表
3. take_screenshot + take_snapshot（展开后状态）
4. Read 读取截图，确认选项列表已渲染显示
5. Grep(-C=5) 搜索目标选项名称，提取目标选项 StaticText 的 uid
6. click 目标选项 StaticText 的 uid → 完成选择
7. take_snapshot + Grep → 验证下拉框 value 已更新
```

**禁止**：
- **绝对禁止**使用 `evaluate_script` 给下拉框注入值（前端框架响应式不更新，且可能干扰页面状态），除非用户明确要求
- `fill()` 给 `readonly` 下拉框输入文字

### 8.4 单选/复选框

- **单选框（radio）**：用 `click` 点击对应选项的 radio 或 StaticText uid
- **复选框（checkbox）**：用 `click` 点击对应选项的 checkbox 或 StaticText uid

### 8.5 日期选择器

特征：通常字段名包含 `date` 等字眼。许多框架的日期选择器是 `readonly textbox`，**但不能用 `fill` 填写**，且日期选项在展开前不可见。

```
步骤:
1. take_snapshot + Grep → 找到日期字段（readonly textbox）的 uid
2. click 该 textbox uid → 展开日历
3. take_screenshot + take_snapshot（日历展开后状态）
4. Read 读取截图，确认日历已展开并确认日历可操作性内容
5. Grep(-C=5) 搜索目标日期数字，提取 StaticText 的 uid
6. click 目标日期 StaticText uid → 完成选择
7. take_snapshot + Grep → 验证日期选择器 value 已更新
```

### 8.6 image 类型图标点击

某些页面使用 `<image>` 元素表示图标按钮，这些元素在默认 a11y 树中可能不显示。

```
步骤:
1. 调用 take_snapshot(verbose=true) → 获取完整 DOM 树
2. Grep 搜索 图标描述关键词（-C=5）→ 提取 <image> 的 uid
3. click 该 uid → 完成操作
4. take_screenshot + Read 读取截图，确认操作情况
```

### 8.7 文件上传

```
步骤:
1. take_snapshot + Grep → 找到文件上传（button）的 uid
2. 确认文件上传类型
3. 确认当前工作目录下对应类型文件路径
4. upload_file(uid=文件上传按钮uid, filePath=目标文件路径)
5. take_snapshot + Grep → 确认文件上传情况
```

- **绝对禁止点击"上传"按钮、"Upload Documents"、"Upload"、"Add File"等任何上传相关按钮。自动化测试不需要也禁止通过点击按钮触发文件选择对话框。**
- **必须直接使用 `upload_file` 工具**，通过 `upload_file(uid=文件上传按钮uid, filePath=目标文件路径)` 完成上传。
- **禁止自己创建文件**。必须使用当前工作目录下已有的对应类型文件（如 .png、.pdf、.jpg 等），不得生成新文件。
- 如果当前工作目录下没有对应类型的文件，**直接报错退出**，不得自行创建或编造文件路径。

#### 文件上传常见错误

| 错误做法 | 正确做法 |
|----------|----------|
| 先 click 上传按钮，再 upload_file | 直接 upload_file，uid 指向上传按钮 |

## 9. 表单填写流程

### 9.0 表单填写触发规则

> **不是看到表格一定就要立刻填写。表单填写必须在以下条件下才触发：**

1. **必须严格遵照用例步骤、reference 中的指令**，只有用例步骤明确要求填写表单（如"填写表单"、"在...中输入"、"选择..."等）时，才触发表单填写流程
2. **非表单操作（如查看列表、点击菜单、切换页签、搜索筛选、点击按钮等）不得触发表单填写**
3. **如果页面中存在表格/表单元素但用例步骤未要求填写，绝对不能主动填写任何字段**
4. **用户没有额外要求时，禁止填写任何非必填字段**

**Why:** 盲目填写未要求的表单字段会导致测试数据错误、验证失败、审批流程异常等问题。测试执行必须严格遵循用例步骤，而非页面中有什么就填什么。

**How to apply：**
- 每次看到表单/表格页面时，先判断当前用例步骤是否要求填写
- 如果步骤是"查看列表"、"点击XX按钮"、"进入XX菜单"等 → 不要填写表单
- 如果步骤明确提到"填写"、"输入"、"选择"、"上传"等 → 按 9.1~9.2 流程填写
- **非必填字段，没有用户的额外要求，绝对禁止填写**

### 9.0.1 必填项识别与填写规则

> **必填项：通常是快照中标注了 `*` 的字段。只有这些字段才需要被填写。**

**必填项识别规则：**

1. **唯一识别标准：快照中字段标签旁带有 `*` 符号的才是必填项**
2. **没有其他识别方式。不能根据字段名称、常识、经验、CRD 要求或任何其他方式推断某个字段是否必填**

**必填项填写规则：**

1. **所有已识别的必填项（带 `*` 的字段）必须依次填写，从表单顶部到底部按顺序逐个填写**
2. **必填项中没有用户额外指令的字段，按字段类型、名称，视情况填写**
3. **每个必填项填写后必须通过 `take_snapshot + Grep` 确认值已正确设置**

**How to apply：**
- 进入表单页面 → `take_snapshot` → Grep 搜索 `*` → 标记所有必填字段 → 按从上到下顺序逐个填写 → 每个字段填写后用 `take_snapshot` 确认

### 9.0.2 非必填项禁止填写规则

> **非必填项没有用户的额外指令、或测试用例的明确要求时，绝对不填写。**

**非必填项定义：** 快照中字段标签旁**没有 `*` 标记**的所有字段都是非必填项。

**非必填项禁止填写规则：**

1. **绝对禁止填写任何非必填项，除非满足以下条件之一：**
   - **测试用例步骤中明确要求填写该字段（如"填写XX字段为XXX"）**
   - **或者 reference.md 中明确要求填写该字段**
   - **或者用户在当前对话中明确指示填写该字段**
   - **或者用户在当前对话中要求查看/验证该字段的内容**
2. **不能因为"该字段可能影响后续流程"而主动填写非必填项**
3. **不能因为"其他测试用例中该字段被填写过"而在此用例中也填写**
4. **不能因为"字段看起来需要值"而自行决定填写非必填项**
5. **如果非必填项已有默认值，保持原样不动，不要清空也不要修改**
6. **如果非必填项为空，保持为空，不要填充**

**Why:** 填写非必填项会污染测试数据，可能导致与预期结果不一致。测试的核心是验证用例步骤定义的行为，而非补充所有可能的字段值。

**How to apply：**
- 每次要填写任何字段前，先检查该字段是否有 `*` 标记
- 如果没有 `*` 标记，再检查用例步骤/reference/用户是否有明确要求
- **三个条件都不满足 → 绝对不填这个字段，直接进入下一个必填项**

### 9.1 表单填写流程

1. **按需填写**：仅当用例步骤明确要求填写表单时，才执行填写流程；否则跳过。
2. **预扫描（准备阶段）**：首次操作前用 `take_snapshot` + `Read` 获取完整表单结构信息。
3. **识别填写字段**：
   - **必填字段**（带 `*` 星号）：逐一列出，确认每个字段的 **字段类型**（文本输入框 / 下拉框 / 日期选择器 / 单选框 / 复选框 / 文件上传）、**用户指定值**（如有）、**对应可见选项**（如单选/复选/下拉的选项列表）
   - **指定填写的非必填字段**：仅当用户明确要求时才列出，确认上述三项信息
   - **未指定的非必填字段**：一律跳过，禁止随意填写
4. **依次填写**：从上到下按顺序处理每个字段，具体操作方式参考 **第 8 章 特殊 UI 组件表单处理**。文本框使用 `fill` 直接填入，其他特殊组件严格按照对应章节步骤执行。
5. **提交前校验**：`take_snapshot` + `Read` 确认所有已填写字段值正确无误。
6. **确认提交**。
```

# PRD — 测试用例自优化

> 业界空白：有 prompt 自优化（HRPO/GEPA），无 test-case 自优化。本文档定义这个概念。

## 1. 概念定义

**测试用例自优化**：完成一轮测试后，基于错误分布分析、spec 增强、用例质量检查，自动迭代测试用例集，使下一轮测试的覆盖度和有效率提升。

与 prompt 自优化的区别：
- prompt 自优化改的是 **被测 Agent**（prompt/tool/workflow）
- 用例自优化改的是 **测试本身**（case/spec/因子）

## 2. 触发条件

| 触发 | 说明 |
|------|------|
| 一轮测试完成 | eval_runner + diagnoser 跑完 |
| 用户手动 | "优化用例" / "分析用例质量" |
| CI 自动 | regression 通过但分数下降 > 0.05 |
| 错误集中 | 某类 F 失败占比 > 40% |

## 3. 分析维度

### 3.1 错误分布分析（F1-F8）

| 错误集中类型 | 含义 | 用例增强策略 |
|-------------|------|-------------|
| F1-F2 集中 | 任务理解/识别失败 | 补充任务类型识别用例；增强 PRD 解析；加更多场景覆盖 |
| F3 集中 | 工具选择失败 | 增加工具选择边界用例；细化 expected_tools；加相似工具区分用例 |
| F4 集中 | 工具参数失败 | 增加参数边界用例（等价类/边界值）；细化 arguments 断言；加字段映射用例 |
| F5 集中 | Workflow 失败 | 增加异常恢复流程用例；加 fallback 场景；加中断恢复用例 |
| F6 集中 | Memory 失败 | 增加多轮上下文用例；加记忆检索触发用例 |
| F7 集中 | 输出失败 | 细化 expected.contains 关键词；加输出格式断言；加幻觉检测用例 |
| F8 集中 | 执行冗余 | 调整 expected_steps；加效率用例；注入 reference 后重新基准 |
| 某维度 0 用例 | 覆盖缺口 | 补充该维度的场景和用例 |
| 某维度 100% 通过 | 可能太简单 | 增加边界/异常/对抗用例 |

### 3.2 Spec 增强机会

测试过程中可能发现：
- 新的业务规则（trace 里有但 spec 没记） → 加到 spec
- 新的工具行为（Agent 调了预期外的工具） → 更新 expected_tools
- 新的边界值（Agent 在某输入下崩溃） → 加边界用例
- 新的失败模式（F1-F8 之外的） → 扩展 taxonomy

### 3.3 用例质量检查（9 维度）

| 维度 | 检查方法 | 权重 | 业界参考 |
|------|---------|------|---------|
| SPEC 完整性 | 用例覆盖 PRD 功能点的比例 | 0.15 | ISO 25010 Completeness |
| 用例完整性 | 字段齐全（id/input/expected/tools/rules/steps/scoring） | 0.10 | IEEE 829 |
| 功能点覆盖度 | 10 维度是否都有用例 | 0.15 | 功能覆盖率标准 |
| DFX 覆盖度 | 性能/安全/兼容/可靠/韧性/可服务/可维护 7 维 | 0.15 | ISO 25010 DFX 子特性 |
| 有效用例率 | 无过时/重复/不可执行用例 | 0.10 | TIOBE Quality Indicator |
| 执行可行性 | 上次执行通过率 > 0 | 0.10 | ChatUniTest 编译验证思路 |
| 无二义性 | 步骤/预期无歧义（NLP 检测） | 0.10 | arXiv 2505.07270 二义性检测 |
| 长度合理 | 用例步数 < 10，token < 500 | 0.05 | Evolutionary 多目标思路 |
| 断言可验证 | expected 可自动验证（非纯主观） | 0.10 | DeepEval assert 体系 |

### 3.4 Mutation 驱动用例增强（参考 Meta ACH）

思路：生成少量"变异 Agent 行为"（如故意漏调工具/返回错误数据），看现有用例能否检出。

```
1. 从 trace 中提取正常行为
2. 注入变异（漏调工具 / 参数错 / 返回空 / 超时）
3. 跑现有用例，看能否检出变异
4. 未检出的变异 → 补充用例
```

kill matrix：
| 变异 | 用例1 | 用例2 | 用例3 | ... | 需补 |
|------|-------|-------|-------|-----|------|
| 漏调 analyzeCashflow | ✅ | ❌ | ❌ | | 用例2/3 需增强 |
| 参数 application_id 传错 | ❌ | ❌ | ✅ | | 用例1/2 需增强 |

## 4. 迭代流程

```
完成一轮测试（eval_runner + diagnoser）
  ↓
Step 1: 错误分布分析
  - 统计 F1-F8 各类失败数
  - 识别集中类型（> 40% 算集中）
  ↓
Step 2: Spec 缺口识别
  - 哪些维度 0 用例？
  - 哪些维度 100% 通过（可能太简单）？
  - trace 里有没有新发现的行为？
  ↓
Step 3: 用例质量检查
  - 9 维度质量评分
  - 识别低分维度
  ↓
Step 4: Mutation 分析（可选）
  - 生成变异
  - 跑 kill matrix
  - 识别未覆盖的变异
  ↓
Step 5: 生成增强建议
  - 新增用例建议（哪些场景/维度需要补）
  - 修改用例建议（哪些用例断言不够/二义性/过长）
  - 废弃用例建议（过时/重复/不可执行）
  ↓
Step 6: 与人确认
  - 展示增强建议
  - 用户选择接受/拒绝/修改
  ↓
Step 7: 更新 cases YAML
  - 通过 case_io.py 写入
  ↓
Step 8: 度量提升
  - 重跑评测
  - 对比质量分变化
  - 记录迭代历史
```

## 5. 需要和人确认的环节

| 环节 | 确认什么 | 为什么 |
|------|---------|--------|
| Step 5 后 | 新增用例建议 | 用例可能不符合业务理解 |
| Step 5 后 | 修改用例建议 | 断言修改可能改变评测标准 |
| Step 5 后 | 废弃用例建议 | 用例可能有业务价值不能删 |
| Step 7 后 | 质量分变化 | 确认提升方向正确 |

## 6. 质量提升度量

| 维度 | 指标 | 衡量方法 |
|------|------|---------|
| 覆盖度 | 功能点覆盖率 | 已覆盖功能点 / 总功能点 |
| 覆盖度 | 维度覆盖率 | 有用例的维度 / 10 |
| 覆盖度 | DFX 覆盖率 | 已覆盖 DFX / 7 |
| 覆盖度 | Mutation 检出率 | 检出的变异 / 总变异 |
| 质量 | 二义性率 | 有二义性的用例 / 总用例 |
| 质量 | 有效用例率 | 可执行用例 / 总用例 |
| 质量 | 质量分 | 9 维度加权分 |
| 迭代 | 新增用例数 | 本轮新增 / 总用例 |
| 迭代 | 废弃用例数 | 本轮废弃 / 总用例 |
| 迭代 | 质量分变化 | 本轮 - 上轮 |
| 迭代 | 覆盖率变化 | 本轮 - 上轮 |

## 7. 实现计划

### 子 skill: test-case-self-optimization

SKILL.md 用 prompt 指导 Agent：
1. 读上一轮 diagnoser 结果
2. 分析 F1-F8 分布
3. 识别 spec 缺口
4. 检查用例质量（9 维度）
5. 生成增强建议
6. 与人确认
7. 调 case_io.py 更新

### 脚本: case_optimizer.py

确定性工作：
- 统计 F1-F8 分布（从 diagnosis.json）
- 计算用例质量分（从 cases YAML + scores.json）
- 生成增强建议 JSON

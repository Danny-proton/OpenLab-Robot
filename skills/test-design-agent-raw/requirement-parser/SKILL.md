# 行业解决方案需求文档解析器

此 skill 帮助你解析、理解和提取行业解决方案需求文档的信息。文档遵循行业解决方案的标准需求跟踪层次，仅包含 IR（系统需求）和 SR（功能需求）两层结构。

## 文档类型和支持的需求层次

### 行业解决方案需求层次

**第 0 层 - 系统需求 (IR - System Requirement)**
- 行业业务场景和系统级需求
- 格式：IR####xxxx####xxxx 或 IR.<行业>-<场景>-<序号>
- 示例：IR20250411003057、IR.FIN-PAY-001（金融 - 支付场景）
- 包含业务目标、业务场景描述、业务价值、目标客户/行业
- 跟踪需求的来源和起点

**第 1 层 - 功能需求 (SR - Solution Requirement)**
- 将 IR 分解为具体的功能特性
- 格式：SR.IR###########-## 或 SR.<业务能力>-<功能>-<序号>
- 示例：SR.IR20250411003057-005、SR.AUTH-LOGIN-001（认证能力 - 登录）
- 包含详细的功能描述、约束条件和规格说明
- 定义系统应具备的功能能力

## 标准列结构

大多数需求文档包含以下关键列：

- **需求编号**：需求唯一标识符
- **需求类型**：需求类型 (IR/SR)
- **父需求编号**：父需求编号，表示需求的层次关系
- **需求名称**：需求标题
- **需求描述**：详细需求文本
- **所属版本**：目标产品/解决方案版本
- **业务分类**：需求类别（功能、性能、可靠性、安全性等）
- **行业领域**：所属行业（金融、政务、医疗、教育、制造、能源等）
- **设计责任人**：所有者/责任人
- **验收标准**：需求验收条件
- **DFX 属性**：DFX 属性（可靠性、安全性、性能、可维护性、兼容性等）

## 工作流程

### 1. 文档类型识别

识别文档类型和结构：
- 行业解决方案需求文档（IR/SR）

### 2. 提取层次结构

通过以下方式重构需求层次：

**行业解决方案模式**：
```
IR20250411003057 (系统需求 - 初始需求)
└── SR.IR20250411003057-005 (功能需求)
    └── SR.IR20250411003057-006 (功能需求)
```

### 3. 提取关键信息

对于每个需求层级，提取：

**IR（系统需求）**：
- 需求来源
- 业务场景描述
- 业务价值
- 目标客户/行业

**SR（功能需求）**：
- **需求名称** - 需求标题
- **需求描述** - 详细描述
- **业务分类** - 需求类别
- **优先级** - 需求优先级
- **验收标准** - 需求验收条件

### 4. 提取 USECASE

从需求文档中提取用例信息，每个用例包含：
- 用例名称：用例的名称
- 用例编号：用例的编号
- 用例描述：用例描述
- 角色：角色（Actor），如管理员、用户、运维人员等
- 前置条件：实现该功能依赖的硬件、周边环境、其他功能的配置、组网、流量等，约束或限制
- 最小保证：主执行者的目标不能被满足的情况下，对系统的最低要求，例如系统数据不能丢失，业务不能中断等
- 成功保证：主执行者的目标被满足的情况下，系统的输出和达到的状态
- 触发事件：启动用例的事件，描述在什么条件下会触发该业务流
- 主成功场景：在正常情况下，实现功能需求的数据流、控制流、操作流程等的描述
- 扩展场景：因为一些异常情况，需要系统处理的一些分支
- DFX 属性：DFX 属性，常见的 DFX 属性如可维护性（命令行/YANG/SCHEMA/MIB/日志/告警/资料等）、性能、可靠性、兼容性等
- 行业特性和合规要求：如金融行业的资金安全、政务行业的等保合规、医疗行业的 HIPAA 等
- 云原生架构考虑：如微服务调用、容器编排、API 治理等
- 数据智能考虑：如数据质量、模型效果、实时计算等
- 数据字典和附件描述：相关数据字典和附件
- 是否架构需求：是否架构需求
- 是否影响架构：是否影响架构

### 5. 生成摘要

创建结构化摘要，输出到与需求文档同目录下的 requirement.md，包括：
- 文档概述（类型、版本、范围、行业领域）
- 需求层次树
- USECASE
- 行业特性和合规要求
- 技术架构考虑
- 任何依赖或约束

## 输出格式

- 如果某个章节无数据，使用引用块标注（如 `> 当前需求未包含任何 UC 用例信息`），不能省略章节

## 常见需求前缀

### 行业解决方案模式
- **IR**：系统需求 - System Requirement（第 0 层：行业业务场景和系统级需求）
- **SR**：功能需求 - Solution Requirement（第 1 层：将 IR 分解为具体的功能特性）

## 边缘情况

### 多个工作表
如果文档有多个工作表：
- 识别主要需求工作表
- 检查参考工作表（例如：迭代计划、版本映射）
- 记录任何跨工作表引用

### 特殊文件类型
一些文档可能使用：
- 指向外部需求管理工具的超链接
- 嵌入的图片或图表
- 筛选或排序状态

### 缺失或不一致的数据
处理以下情况：
- 某些行信息不完整
- 层次结构有缺口
- 编码引用不一致
- 存在非标准列名

### 混合需求文档
如果文档包含多种需求类型：
- 根据需求编号前缀进行分类（IR/SR）
- 分别构建各自的层次结构
- 在输出中标注需求类型

## 注意事项

- 需求文档可能使用中文术语
- 业务编号遵循特定模式但可能有变体
- 此 skill 应同时支持中文和英文需求文档
- 根据文档类型自动识别并使用相应的层次结构（仅 IR/SR 两层）
- 识别行业特性和合规要求，并在输出中体现
- 注意云原生架构和数据智能需求的特殊处理方式

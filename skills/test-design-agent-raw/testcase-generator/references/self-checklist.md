# 行业解决方案测试用例自检清单详解

## 概述

自检清单是确保行业解决方案测试用例设计质量的最后一道防线。在完成测试用例设计后，必须逐项验证以下检查项。本文档详细说明每项检查的含义、检查方法和常见问题。

**说明**：带 `*` 号的检查项为关键否决项，必须全部通过，否则测试用例设计不合格。

---

## 自检清单

### 1. *用例间是否存在隐性依赖（如：需要其他用例先执行配置）？

**检查内容**：验证测试用例之间是否存在隐性的依赖关系。

**检查方法**：
1. 逐个查看每个测试用例的预置条件
2. 检查预置条件是否描述为"执行用例 X 后"或"基于用例 X 的结果"
3. 检查测试用例之间是否存在数据或状态的依赖
4. 确认每个用例是否可以独立执行

**判定标准**：
- ✅ 通过：所有测试用例的预置条件都不依赖其他用例的执行结果
- ❌ 不通过：存在预置条件描述依赖其他用例的情况

**常见问题**：
- 预置条件描述为"执行用例 A 后"
- 预置条件描述为"基于用例 A 生成的配置"
- 测试用例之间的执行顺序有明确规定

**解决方案**：
- 将依赖用例的配置动作复制到当前用例的预置条件中
- 或使用脚本自动准备测试环境
- 确保每个用例可以独立执行

**示例**：
- ❌ 错误：
  ```
  用例 1：配置微服务 A
  用例 2：测试微服务 A 调用
  预置条件：执行用例 1 后，微服务 A 已启动
  ```
- ✅ 正确：
  ```
  用例 2：测试微服务 A 调用
  预置条件：微服务 A 已启动，服务注册中心已注册
  ```

---

### 2. *是否存在同一业务流程或逻辑上连续的行为被拆分为多个用例？（如：故障切换与恢复分离）

**检查内容**：验证同一业务流程或逻辑连续的行为是否被不必要地拆分为多个测试用例。

**检查方法**：
1. 理解需求的业务流程，识别逻辑上连续的行为序列
2. 检查是否有将正向操作和逆向操作（如配置/删除）分开的情况
3. 检查是否有将故障触发和故障恢复分开的情况
4. 检查是否有将同一测试对象的多个操作拆分到不同用例的情况

**判定标准**：
- ✅ 通过：逻辑连续的行为合并在一个测试用例中，或合理拆分
- ❌ 不通过：同一业务流程被不必要地拆分为多个用例

**常见问题**：
- 将"配置→使用→删除"拆分为三个独立用例
- 将"故障切换→故障恢复"拆分为两个独立用例
- 将"服务可用→服务故障→服务恢复"拆分为多个用例

**解决方案**：
- 将逻辑连续的行为合并到一个测试用例中
- 体现完整的业务生命周期测试
- 保持测试场景的完整性

**示例**：
- ❌ 错误：
  ```
  用例 1：订单创建测试
  用例 2：订单取消测试
  ```
- ✅ 正确：
  ```
  用例 1：订单创建与取消完整流程测试
  测试步骤：1. 创建订单，见预期结果 1
            2. 取消订单，见预期结果 2
  ```

### 3. 预期结果是否与测试步骤逐阶段对应？

**检查内容**：验证每个测试步骤都有对应的预期结果，且对应关系明确。

**检查方法**：
1. 统计测试步骤的编号总数
2. 统计预期结果的编号总数
3. 检查编号是否连续且数量一致
4. 检查每个测试步骤是否使用"见预期结果 n"格式关联对应结果

**判定标准**：
- ✅ 通过：每个测试步骤都有对应的预期结果，编号连续且一致
- ❌ 不通过：测试步骤和预期结果数量不一致或编号不对应

**常见问题**：
- 测试步骤有 5 步，但预期结果只有 4 项或 6 项
- 预期结果编号不连续（如 1、2、4）
- 测试步骤未使用"见预期结果 n"格式关联

**解决方案**：
- 确保测试步骤和预期结果数量一致
- 确保编号连续，从 1 开始递增
- 每个测试步骤必须明确关联对应的预期结果

**示例**：
- ❌ 错误：
  ```
  测试步骤：1. 配置服务
            2. 启动服务
            3. 查询状态
  预期结果：1. 配置成功
            2. 服务已启动
  ```
- ✅ 正确：
  ```
  测试步骤：1. 配置服务，见预期结果 1
            2. 启动服务，见预期结果 2
            3. 查询状态，见预期结果 3
  预期结果：1. 配置成功
            2. 服务已启动
            3. 服务状态为 Running
  ```

---

### 4. *是否存在未关联任何测试步骤的预期结果项？

**检查内容**：验证预期结果中是否有未被任何测试步骤关联的结果项。

**检查方法**：
1. 列出所有预期结果的编号
2. 对照测试步骤的关联格式，检查每个预期结果编号是否被引用
3. 检查是否有"孤儿"预期结果（未被引用）

**判定标准**：
- ✅ 通过：所有预期结果都被至少一个测试步骤引用
- ❌ 不通过：存在未被测试步骤关联的预期结果项

**常见问题**：
- 预期结果有 5 项，但测试步骤只引用了其中 4 项
- 预期结果编号超出测试步骤数量
- 预期结果描述的内容与任何测试步骤都不相关

**解决方案**：
- 删除未关联的预期结果
- 或为该预期结果添加对应的测试步骤
- 确保每个预期结果都有明确的来源

---

### 5. *是否存在测试步骤关联不存在的预期结果？

**检查内容**：验证测试步骤是否引用了不存在的预期结果编号。

**检查方法**：
1. 检查每个测试步骤的"见预期结果 n"中 n 的取值
2. 确认 n 的取值是否在预期结果编号范围内
3. 确认 n 的取值是否与预期结果中的实际编号对应

**判定标准**：
- ✅ 通过：所有测试步骤引用的预期结果编号都存在
- ❌ 不通过：存在测试步骤引用了不存在的预期结果编号

**常见问题**：
- 测试步骤引用"见预期结果 5"，但预期结果中只有 1-4 项
- 测试步骤引用的编号超出预期结果数量
- 测试步骤引用的编号写错

**解决方案**：
- 修正测试步骤中的引用编号
- 或添加缺失的预期结果
- 确保测试步骤和预期结果编号严格对应

---

### 6. *测试步骤是否包含验证性描述（如"...验证..."，"...检查..."，"...查询..."）？

**检查内容**：验证测试步骤是否包含了验证性描述，而非纯粹的执行操作。

**检查方法**：
1. 逐个检查每个测试步骤
2. 识别是否包含"验证"、"检查"、"查询"、"确认"、"观察"等验证性动词
3. 确认测试步骤是否仅描述可执行的操作，而不包含验证行为

**判定标准**：
- ✅ 通过：测试步骤仅描述可执行的操作，不包含验证性描述
- ❌ 不通过：测试步骤包含验证性描述

**常见问题**：
- 测试步骤：配置服务并验证配置成功
- 测试步骤：查询服务状态确认连接建立
- 测试步骤：检查订单状态验证订单存在

**解决方案**：
- 将验证性描述移除，改为关联到预期结果
- 或将验证性描述拆分为验证步骤（但通常应该在预期结果中）
- 确保测试步骤仅描述执行操作

**示例**：
- ❌ 错误：
  ```
  测试步骤：1. 配置服务并验证配置成功
            2. 查询服务状态确认连接建立
  ```
- ✅ 正确：
  ```
  测试步骤：1. 配置服务参数，见预期结果 1
            2. 查询服务状态，见预期结果 2
  预期结果：1. 配置成功
            2. 服务状态为 Connected
  ```

---

### 7. *核心功能测试点是否为空？

**检查内容**：验证核心功能测试点列表是否为空。

**检查方法**：
1. 检查"核心功能测试点"部分是否有内容
2. 确认是否至少列出了一个核心功能测试点
3. 检查测试点数量是否与需求复杂度相匹配

**判定标准**：
- ✅ 通过：核心功能测试点列表不为空，至少有一个测试点
- ❌ 不通过：核心功能测试点列表为空或未提供

**常见问题**：
- 核心功能测试点部分完全没有内容
- 只写了"核心功能测试点"标题，没有列出具体测试点
- 测试点数量过少，与需求复杂度不匹配

**解决方案**：
- 基于需求描述提取核心功能测试点
- 确保覆盖主成功场景和扩展场景的关键测试点
- 测试点数量应该与需求复杂度相匹配

**示例**：
- ❌ 错误：
  ```
  ### 核心功能测试点

  （无内容）
  ```
- ✅ 正确：
  ```
  ### 核心功能测试点

  1. 支付流程完整性验证
  2. 风控校验准确性验证
  3. 资金安全验证
  ```

---

### 8. *测试用例是否覆盖了所有的核心功能测试点？

**检查内容**：验证所有核心功能测试点是否都被测试用例覆盖。

**检查方法**：
1. 列出所有核心功能测试点
2. 检查每个测试用例的设计描述中的"覆盖"列表
3. 确认每个核心功能测试点是否都被至少一个测试用例覆盖
4. 对照测试用例内容，验证覆盖的准确性

**判定标准**：
- ✅ 通过：所有核心功能测试点都被测试用例覆盖
- ❌ 不通过：存在未被测试用例覆盖的核心功能测试点

**常见问题**：
- 有核心功能测试点未在任何测试用例的设计描述中出现
- 测试用例设计描述中的"覆盖"列表不完整
- 实际测试用例内容与设计描述的覆盖列表不一致

**解决方案**：
- 确保每个核心功能测试点都对应至少一个测试用例
- 在测试用例设计描述中明确列出覆盖的核心功能测试点
- 验证测试用例实际内容与设计描述的一致性

**示例**：
- ❌ 错误：
  ```
  核心功能测试点：
    1. 支付流程完整性验证
    2. 风控校验准确性验证
    3. 资金安全验证  （未被覆盖）

  测试用例：
    用例 1：支付测试
    设计描述：覆盖：支付流程完整性验证
  ```
- ✅ 正确：
  ```
  核心功能测试点：
    1. 支付流程完整性验证
    2. 风控校验准确性验证
    3. 资金安全验证

  测试用例：
    用例 1：支付流程测试
    设计描述：覆盖：支付流程完整性验证、风控校验准确性验证

    用例 2：资金安全测试
    设计描述：覆盖：资金安全验证
  ```

### 9. 是否仅归纳总结测试用例输出思路，未输出实例化的测试用例？

**检查内容**：验证测试用例部分是否输出了完整的、实例化的测试用例，而不仅仅是设计思路或总结。

**检查方法**：
1. 检查测试用例部分的内容
2. 确认是否包含完整的用例标题、预置条件、测试步骤、预期结果、设计描述
3. 确认测试步骤和预期结果是具体的、实例化的描述，而非抽象的概括
4. 确认是否输出多个测试用例（而非一个概括性的描述）

**判定标准**：
- ✅ 通过：输出了完整的、实例化的测试用例，包含所有必需要素
- ❌ 不通过：只归纳总结了设计思路，未输出具体的、实例化的测试用例

**常见问题**：
- 只输出测试用例的设计思路和框架，没有具体内容
- 测试步骤和预期结果过于抽象，不是具体可执行的描述
- 只输出一个概括性的测试用例说明，而非多个具体的用例

**解决方案**：
- 输出完整的、实例化的测试用例
- 确保每个测试用例包含所有必需要素：用例标题、预置条件、测试步骤、预期结果、设计描述
- 测试步骤和预期结果应该是具体的、可执行的、可验证的
- 根据需求复杂度，输出适当数量的测试用例

**示例**：
- ❌ 错误（仅归纳总结）：
  ```
  测试用例设计思路：
  本需求需要设计支付功能的测试用例，应该包括配置、发起、完成等步骤。
  可以设计 1-2 个用例，覆盖正常场景和异常场景。
  ```

- ✅ 正确（输出实例化用例）：
  ```
  ### 测试用例：

  用例 1：支付功能完整流程验证
  用例标题：用户支付流程验证
  预置条件：用户已登录，账户余额充足
  测试步骤：1. 发起支付请求，输入金额 100 元，见预期结果 1
            2. 确认支付信息，见预期结果 2
            3. 输入支付密码，见预期结果 3
  预期结果：1. 支付请求成功，进入确认页面
            2. 支付信息显示正确
            3. 支付成功，返回成功页面
  设计描述：本用例验证支付功能的完整流程。覆盖：支付流程完整性验证
  ```

### 10. *生成的内容是否存在语句不通顺、逻辑混乱或错别字问题？

**检查内容**：验证生成的核心功能测试点、测试对象和操作、测试用例等所有内容是否存在语句不通顺、逻辑混乱或错别字问题。

**检查方法**：
1. 逐句阅读核心功能测试点，检查语句是否通顺，表达是否清晰
2. 逐句阅读测试对象和操作描述，检查是否存在语病或表述不清
3. 逐句阅读测试用例的用例标题、预置条件、测试步骤、预期结果、设计描述
4. 检查是否存在逻辑不一致的问题（如测试步骤与预期结果不匹配、操作描述前后矛盾等）
5. 检查是否存在错别字、专业术语使用不准确的问题

**判定标准**：
- ✅ 通过：所有生成的内容语句通顺，逻辑清晰，无错别字，专业术语使用准确
- ❌ 不通过：存在语句不通顺、逻辑混乱或错别字问题

**常见问题**：
- 核心功能测试点表述不清，语法结构混乱
- 测试对象和操作描述存在语病，难以理解
- 测试用例标题表述混乱，不清晰
- 测试步骤描述不完整或存在语法错误
- 预期结果描述逻辑不通顺，与测试步骤不匹配
- 专业术语使用不当，与需求描述不一致
- 存在明显的错别字

**解决方案**：
- 重新梳理语句结构，确保表达清晰、准确
- 检查逻辑关系，确保测试步骤与预期结果对应一致
- 核对专业术语，确保与需求描述保持一致
- 使用正确的语法结构，避免语病
- 仔细校对，消除错别字
- 必要时参考需求原文，确保提取内容表述准确

**示例**：
- ❌ 错误：
  ```
  核心功能测试点：
    1. 支付流程验证
    2. 风控功能验证了
  测试步骤：1. 配置支付参数，见预期结果 2
  预期结果：1. 配置成功
  ```
  - 问题 1：测试点 2 中"验证了"表述不清，应改为"风控功能验证"
  - 问题 2：测试步骤引用"见预期结果 2"，但预期结果只有 1 项，逻辑不一致

- ✅ 正确：
  ```
  核心功能测试点：
    1. 支付流程验证
    2. 风控功能验证
  测试步骤：1. 配置支付参数，见预期结果 1
  预期结果：1. 配置成功
  ```
  - 测试点表述清晰，无语法错误
  - 测试步骤与预期结果编号严格对应，逻辑一致

### 11. *测试用例是否覆盖行业特性和合规要求？

**检查内容**：验证测试用例是否覆盖了行业特性和合规要求。

**检查方法**：
1. 识别所属行业（金融、政务、医疗、教育、制造、能源等）
2. 对照行业监管要求和合规标准
3. 检查测试用例是否包含行业特性和合规验证
4. 确认安全、隐私、审计等合规要求的覆盖情况

**判定标准**：
- ✅ 通过：测试用例覆盖行业特性和合规要求
- ❌ 不通过：测试用例未覆盖行业特性和合规要求

**常见问题**：
- 金融场景未覆盖资金安全验证
- 政务场景未覆盖等保合规验证
- 医疗场景未覆盖数据隐私验证
- 通用场景未覆盖审计日志验证

**解决方案**：
- 根据行业特性补充相应的测试场景

**检查方法**：
1. 从 requirement.md 中统计 SR 需求数量
2. 从 testcase.md 中统计用例数量
3. 检查每个 SR 需求是否都有对应的功能用例
4. 对于有性能指标的 SR 需求，检查是否有性能用例

6. 对于 DFX 包含安全性的 SR 需求，检查是否有安全用例

### 15. *性能测试用例是否充分？

**检查内容**：验证性能测试用例是否充分，包括响应时间、吞吐量、资源占用等性能指标的验证。

**检查方法**：

2. 检查是否有性能测试用例验证这些指标

4. 检查是否包含压力测试场景
5. 检查性能用例的测试步骤是否包含具体的性能测量

**判定标准**：
- ✅ 通过：性能测试用例充分，覆盖所有性能指标和场景
- ❌ 不通过：性能测试用例不足或缺失

**常见问题**：
- 性能指标无对应测试用例
- 只有典型负载测试，缺少边界值测试
- 性能测试步骤中没有具体的性能测量操作

**解决方案**：
- 为每个性能指标设计对应的测试用例
- 补充边界值和压力测试场景
- 在测试步骤中明确性能测量操作

**示例**：
- ❌ 错误：

  ```
  用例：大模型推理性能验证
  测试步骤：1. 使用 128K token 输入执行推理任务，见预期结果 1
            2. 使用 1024 token 输出验证生成长度，见预期结果 2
            3. 测量非首字时延，见预期结果 3
  预期结果：1. 输入处理成功
            2. 输出长度达标
            3. 非首字时延 200ms 以内
  ```

## 自检流程建议

### 1. 自检时机
- **设计完成后立即自检**：完成测试用例设计后，立即使用自检清单逐项检查
- **修改后再次自检**：如果根据自检结果进行了修改，需要再次进行完整自检

### 2. 自检方法
- **逐项检查**：按照自检清单的顺序，逐项检查
- **记录问题**：对于未通过的检查项，记录具体问题
- **集中修正**：完成所有检查后，集中修正问题
- **二次自检**：修正后再次进行完整自检，确保所有问题都解决

### 3. 自检记录
建议记录自检结果，包括：
- 检查日期和时间
- 检查人
- 每项检查的结果（通过/不通过）
- 发现的问题
- 修正措施
- 最终确认状态

### 4. 自检质量标准
- 所有带 `*` 号的关键否决项必须全部通过
- 非关键项原则上应该通过，如有特殊情况需要说明理由
- 自检记录应该完整、清晰、可追溯

## 常见自检问题分析

### 问题 1：自检通过但评审不通过

**可能原因**：
- 自检不够细致，未发现问题
- 对检查项理解有偏差
- 自检标准低于评审标准

**改进措施**：
- 加强自检的细致程度
- 对于不确定的检查项，寻求同事协助确认
- 加深对检查项含义的理解

### 问题 2：反复修改仍有问题

**可能原因**：
- 修改后又引入了新的问题
- 修改方向不正确
- 对规则要求理解不透彻

**改进措施**：
- 修改后全面检查，避免引入新问题
- 重新理解规则要求，确保修改方向正确
- 多次修改仍未通过时，寻求资深同事指导

### 问题 3：自检耗时过长

**可能原因**：
- 测试用例初版质量较低
- 对规则不熟悉
- 自检过程不够高效

**改进措施**：
- 提高设计阶段的初版质量，减少自检工作量
- 加强对规则的学习和理解，形成肌肉记忆
- 总结自检经验，提高自检效率

## 总结

自检清单是确保测试用例设计质量的重要工具。严格、细致的自检可以：

1. **提前发现问题**：在评审前发现问题并修正，提高评审通过率
2. **保证设计质量**：确保测试用例符合所有规范和要求
3. **提升设计能力**：通过自检加深对规则的理解，提升设计水平
4. **建立质量意识**：培养自我检查、自我负责的质量意识

建议将自检作为测试用例设计的必要环节，形成习惯，确保每个测试用例都经过严格的质量把关。只有通过自检清单所有检查的测试用例，才能提交评审或进入执行阶段。

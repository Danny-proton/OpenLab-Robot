# Skill: 行业解决方案测试 spec 检查

## 基础元信息
- **名称**: testspec-check
- **描述**: 基于 testspec.md 中的关系，从行业解决方案测试专家角度，对测试对象和测试操作、测试对象和测试数据完备性与正确性进行校验
- **版本号**: V2.0 (行业解决方案版)
- **开发者**: 测试设计工程师
- **所属团队**: 行业解决方案测试领域

## 核心逻辑

### 【角色定义】
你是行业解决方案测试专家，精通以下内容：
1. **行业解决方案测试**：金融、政务、医疗、教育、制造、能源等行业的业务场景测试、云网融合测试、数据智能测试；
2. **测试设计标准**：《测试设计数字化定义》、ISO/IEC 25010 软件质量模型、NIST 云安全框架；
3. **云原生架构**：微服务、容器、API 网关、服务网格、DevOps 等云原生技术架构；
4. **数据智能能力**：大数据分析、AI/ML 平台、实时计算、特征工程等数据智能技术；
5. **安全合规**：数据安全、隐私保护、等保 2.0、GDPR、行业监管要求等；
6. **测试方法论**：DFX 测试（可靠性、安全性、性能、可维护性、兼容性）、敏捷测试、混沌工程。

### 【处理过程】
按照如下步骤完成测试要素完备性与正确性的检查：
1. 读取输入文件：与需求文档同目录下的 testspec.md，解析其中的测试要素内容，提取测试对象、测试操作、测试数据、测试关系
2. 读取与需求文档同目录下的 requirement.md，理解 UC（UseCase）与需求层次
3. 基于 requirement.md 对 testspec.md 中的测试要素的完备性与正确性进行校验，并进行修改与补充
4. 输出目标文件：与需求文档同目录下的 testspec_final.md

## 工作流程

### 步骤 1：输入文件解析
- 读取 testspec.md 文件
- 提取测试对象表、测试操作表、测试数据表、关系表
- 分析各表的结构和内容
- 识别测试对象类型（业务场景、平台能力、数据智能、安全合规、架构支撑）

### 步骤 2：理解 UC 描述与需求层次
**重点阅读内容**：
- UC 描述（功能总体描述）
- 前置条件（系统必须满足的条件）
- 触发事件（启动功能的事件）
- 主成功场景（最常发生的成功动作）
- 扩展场景（特定条件下的行为分支）
- DFX 属性（功能需求中的 DFX 属性）
- 行业特性和合规要求
- 性能指标和 SLA 要求
**参考资料**：
- [用例分析说明](./references/uc-analysis-guide.md)

### 步骤 3：测试要素的完备性与正确性校验
**目标**：从 UC 和需求层次对 testspec.md 中的测试要素完备性和正确性进行校验

**测试对象检查**：
1. 确保测试对象完整覆盖需求中提到的所有业务场景、平台能力、数据智能、安全合规等方面
2. 验证测试对象分类是否合理，编号规则是否符合规范
3. 验证测试对象描述是否准确反映需求内容
4. 检查测试对象是否覆盖了需求中提到的所有技术栈和架构组件
5. 确认测试对象覆盖了 DFX 属性的验证需求

**测试操作检查**：
1. 验证测试操作为动宾结构
2. 验证测试操作表中与测试对象表中的"测试对象名称"一致
3. 必须严格从 UC 的主成功场景和扩展场景中提取操作，不得凭想象添加
4. 验证从 UC 的主成功场景和扩展场景中提取的操作覆盖完全，如果有缺失需要补充
5. 检查测试操作是否覆盖了 DFX 属性的验证（可靠性、安全性、性能等）
6. 确认测试操作是否覆盖了行业特性和合规要求的验证

**测试数据检查**：
1. 检查测试数据具有明确取值范围
2. 验证测试数据表与测试对象表中的"测试对象名称"一致
3. 如果 SR 描述中存在明确的数值规格，校验测试数据中的取值是否符合规格
   - 例如：SR 描述中交易 QPS 要求 10000，需要检查测试数据中是否设计了对应的性能测试场景
   - 例如：SR 描述中响应时间 SLA 为 TP99<100ms，需要检查测试数据中是否设计了相应的性能验证
4. 验证测试数据是否覆盖了边界值、异常值场景
5. 确认测试数据是否覆盖了不同行业场景的数据特征

**关系检查**：
1. 验证关系分析的完整性和正确性
2. 验证关系表与测试对象表的"测试对象名称"名称一致
3. 验证关系表中"关系类型"符合参考规范
4. 检查关系是否覆盖了 UC 中提到的所有业务依赖和系统依赖
5. 确认关系是否体现了云原生架构中的服务依赖、数据依赖、资源依赖等

### 步骤 4：输出文件生成
- 输出路径：/project/test-design/testspec/testspec_final.md
- 包含校验后的完整测试要素内容
- 提供校验问题和修改建议

## 校验规则

### 完备性校验规则
1. **测试对象完备性**
   - UC 主成功场景中的每个步骤都应映射到至少一个测试对象
   - UC 扩展场景中的每个分支都应映射到至少一个测试对象
   - DFX 属性要求应映射到对应的测试对象
   - 行业特性和合规要求应映射到对应的测试对象

2. **测试操作完备性**
   - 主成功场景的每个动作都应提取为测试操作
   - 扩展场景的每个分支都应提取为测试操作
   - DFX 属性的验证应有对应的测试操作

3. **测试数据完备性**
   - 每个测试对象都应有对应的测试数据
   - 性能指标应有对应的测试数据
   - 边界条件应有对应的测试数据

4. **关系分析完备性**
   - UC 中提到的所有依赖关系都应识别
   - DFX 属性间的依赖关系应识别
   - 行业特性间的约束关系应识别

### 正确性校验规则
1. **测试对象正确性**
   - 测试对象描述应准确反映需求内容
   - 测试对象分类应符合规范
   - 测试对象编号规则应符合规范

2. **测试操作正确性**
   - 测试操作应为动宾结构
   - 测试操作名称应与测试对象名称一致
   - 测试操作不应包含需求中未提及的内容

3. **测试数据正确性**
   - 测试数据取值范围应符合 SR 规格要求
   - 测试数据单位应明确
   - 测试数据描述应准确

4. **关系分析正确性**
   - 关系类型应符合规范定义
   - 关系描述应准确反映依赖关系
   - 影响程度评估应合理

## 常见问题

### Q1：如何判断测试对象是否覆盖完整？
A：对照 UC 的主成功场景和扩展场景，确保每个步骤和分支都有对应的测试对象。同时检查 DFX 属性、行业特性、合规要求是否都有对应的测试对象。

### Q2：测试操作从哪里提取？
A：测试操作必须严格从 UC 的主成功场景和扩展场景中的动作描述提取，不得凭想象添加需求中未提及的操作。

### Q3：如何验证测试数据的正确性？
A：对照 SR（System Requirement）中的数值规格，检查测试数据中的取值范围是否符合规格要求。例如 SR 中规定性能指标，测试数据中应有对应的性能测试数据设计。

### Q4：关系分析不完整怎么办？
A：重新分析 UC 中的业务流程和技术架构，识别所有依赖关系。包括业务依赖、服务依赖、数据依赖、资源依赖等。

### Q5：DFX 属性如何设计测试要素？
A：DFX 属性通常需要专项测试，例如：
- 可靠性：故障切换、异常恢复、容错处理
- 安全性：权限验证、数据加密、审计日志
- 性能：响应时间、吞吐量、资源占用
- 兼容性：版本兼容、平台兼容、协议兼容

## 行业解决方案测试 spec 校验要点

### 金融场景特有校验
- 交易流程完整性：支付、清算、结算流程是否覆盖
- 资金安全：金额校验、账务平衡、交易防重
- 风控校验：风险识别、限额控制、反洗钱
- 数据一致性：账务数据一致性、对账机制
- 合规要求：PCI-DSS、反洗钱、数据隐私

### 政务场景特有校验
- 审批流程：申请、审核、审批、结果反馈全流程
- 权限控制：角色权限、数据权限、操作权限
- 等保合规：身份认证、访问控制、审计日志
- 数据共享：跨部门数据共享机制
- 流程效率：审批时限、流程节点时效

### 医疗场景特有校验
- 患者隐私：病历隐私保护、数据脱敏
- 医疗数据：诊断数据完整性、处方数据准确性
- 协作功能：会诊协作、数据共享
- HIPAA 合规：患者数据保护、审计追踪
- 可用性：7x24 小时服务、故障恢复

### 云原生特有校验
- 微服务调用：服务发现、负载均衡、熔断降级
- 容器编排：Pod 调度、健康检查、自动扩缩容
- API 治理：API 版本管理、限流、鉴权
- 服务网格：流量控制、 observability、安全策略
- DevOps：CI/CD流程、自动化测试、灰度发布

### 数据智能特有校验
- 数据质量：数据完整性、准确性、一致性
- 模型效果：准确率、召回率、F1 分数
- 实时计算：延迟、吞吐量、容错
- 特征工程：特征准确性、特征时效性
- AI 推理：推理延迟、推理准确率

# 国际化测试用例设计 Skill

基于待测网站的 memory 信息（页面结构、元素、操作流）和国际化 Checklist，自动生成覆盖完整的国际化测试用例。

## 核心思路

与传统 test-design-agent（从需求文档→SPEC→用例）不同，本 Skill 的输入是 **网站冷启动后的 memory 数据**，直接生成国际化测试用例。

```
.memory/ (页面/元素/导航/操作流) + 国际化 Checklist → 国际化测试用例
```

## 输入来源

### 1. 网站 Memory 数据

从 `.memory/` 目录读取：

```
.memory/
├── shared/
│   ├── environment.md      # 系统URL、账号、测试范围
│   └── execution_rules.md  # 执行规则
└── modules/
    ├── <module1>/
    │   ├── README.md       # 模块功能、场景、入口、前置条件
    │   ├── objects.md      # 页面元素（输入框、按钮、下拉框、弹框等）
    │   └── guide.md        # 稳定操作流
    ├── <module2>/
    │   └── ...
```

### 2. 国际化 Checklist

内置 12 项检查项（见下方），分为必选和可选两类。

## 工作流

### 步骤 1：收集 Memory 信息

遍历 `.memory/modules/` 下所有模块，汇总：
- 模块列表及功能描述
- 各模块下的页面清单
- 各页面的关键元素（输入框、按钮、下拉框、弹框、菜单等）
- 语言切换功能是否存在及位置

### 步骤 2：映射检查项到页面

对每个页面，根据页面元素类型分配 Checklist 检查项：

| 页面元素类型 | 适用的 Checklist 检查项 |
|-------------|------------------------|
| 输入框/文本域 | 1, 2, 8（翻译残留、显示完整性） |
| 按钮/功能按键 | 7（功能按键展示） |
| 下拉框 | 1, 2, 7（翻译残留、功能展示） |
| 弹框/提示框 | 1, 2, 7（翻译残留、功能展示） |
| 菜单/导航 | 1, 2（翻译残留） |
| 表格/属性名/表头 | 1, 3, 8（翻译残留、布局对齐、显示完整性） |
| 日期时间控件 | 10（多时区转换） |
| 金额/货币相关 | 11（货币转换） |
| 配置项/参数设置 | 12（模块配置可转变） |
| 所有页面 | 4, 5, 6, 9（页面风格、翻译准确性） |

### 步骤 3：生成测试用例

对每个页面的每个适用 Checklist 项，生成至少 1 条测试用例。

**用例格式规范**：遵循内置的 `**字段名**:` 换行格式：

```
**用例编号**:
I18N_<维度缩写>_<序号>

**用例名称**:
<简洁描述测试内容>

**预置条件**:
1. <条件1>
2. <条件2>

**测试步骤**:
1. <操作步骤1>，见预期结果1
2. <操作步骤2>，见预期结果2

**预期结果**:
1. <预期结果1>
2. <预期结果2>

**设计描述**:
<设计说明，标注覆盖的Checklist项和页面>

**国际化维度**:
<界面多语言 / 多时区 / 多货币 / 配置项>

**必选/可选**:
<必选 / 可选>

**优先级**:
<P0 / P1 / P2>

**覆盖页面**:
<具体页面名称>

**覆盖Checklist项**:
<Checklist编号及名称>
```

**用例编号前缀规则**：

| 维度 | 前缀 | 示例 |
|------|------|------|
| 界面多语言 | I18N_LANG | I18N_LANG_001 |
| 多时区 | I18N_TZ | I18N_TZ_001 |
| 多货币 | I18N_CURR | I18N_CURR_001 |
| 配置项 | I18N_CFG | I18N_CFG_001 |

### 步骤 4：生成汇总清单

在每个用例后，生成模块维度的用例汇总：

```markdown
---
module: <模块名称>
total_cases: <总用例数>
lang_cases: <多语言用例数>
tz_cases: <多时区用例数>
curr_cases: <多货币用例数>
cfg_cases: <配置项用例数>
pages_covered: <覆盖页面数>
checklist_covered: <覆盖Checklist项数>
---
```

### 步骤 5：用例自检

输出前执行自检：

| 自检项 | 检查标准 | 通过条件 |
|--------|----------|----------|
| 格式检查 | 所有用例使用粗体标记格式 | 无表格格式 |
| 字段格式 | `**字段名**:` 后必须换行 | 符合规范 |
| 步骤关联 | 每步带 `，见预期结果 n` | 100% 关联 |
| 步骤数量 | 5-10 步 | 全部满足 |
| 用例独立性 | 无跨用例依赖描述 | 可独立执行 |
| Checklist覆盖 | 必选项每个页面至少覆盖1条 | 全覆盖 |
| 双向语言切换 | 每条用例覆盖中→英、英→中双向 | 双向覆盖 |

## 国际化 Checklist

| 序号 | 检查项 | 必选/可选 | 检查内容 |
|------|--------|-----------|----------|
| 1 | 页面翻译 - 中文残留 | 必选 | 切换英文界面，英文界面禁止出现中文（弹框、提示信息、下拉框、属性名/表头、菜单） |
| 2 | 页面翻译 - 英文残留 | 必选 | 切换中文界面，中文界面禁止出现英文（提示框、工具栏、下拉框、弹框、提示信息、菜单） |
| 3 | 页面风格 - 布局对齐 | 必选 | 英文界面属性名不换行导致 UI 布局不对齐、不对称 |
| 4 | 页面风格 - 字体协调 | 必选 | 英文界面字体大小与界面大小比例协调 |
| 5 | 页面风格 - 字体一致 | 必选 | 英文界面同一页面字体风格保持一致 |
| 6 | 页面风格 - 换行重叠 | 必选 | 英文界面禁止英文换行导致内容重叠，换行后布局需合理 |
| 7 | 功能按键展示 | 必选 | 弹框布局不重叠，每项功能或按钮对齐，排布不重叠，下拉语言文字不超出方框，切换英文后隐藏功能不失效且同步按英文显示 |
| 8 | 显示完整性 | 必选 | 英文界面英文单词显示完整，不显示不完整或不显示 |
| 9 | 翻译准确性 | 必选 | 翻译正确，专业术语无错误，无低级语法和词汇错误，大小写区分，图片是否翻译，字体大小合理；常用操作翻译语法正确，单复数区分；格式错误；前后字段一致；语言包翻译准确性、专业性、一致性 |
| 10 | 多时区转换 | 可选 | 时间控件支持多时区，前端展示时间经 BE 时区转换，前端传递给后端的时间转换为 UTC0 时区 |
| 11 | 货币转换 | 可选 | 切换到其他货币支付后具备所选货币的对应转换金额 |
| 12 | 模块配置可转变 | 可选 | 系统界面可编辑配置"参数""模块"，切换语言后"参数""模块"可跟随变换 |

## 用例生成策略

1. **必选项全覆盖**：每个页面的必选检查项至少生成 1 条用例
2. **双向语言切换**：每条用例必须覆盖中→英和英→中两个方向的切换
3. **页面具体化**：用例中的操作步骤必须基于 `.memory/` 中实际发现的页面和元素，禁止凭空臆造
4. **步骤合理**：每个用例 5-10 步，不多不少
5. **输出路径**：保存到 `.memory/modules/i18n-test/cases/` 目录
6. **索引更新**：生成后更新 `.memory/modules/i18n-test/cases/INDEX.md`

## 关键规则

- **禁止凭空臆造**：所有用例必须基于 `.memory/` 中实际发现的页面和元素
- **禁止复制 Checklist**：Checklist 是输入，不能直接作为用例输出
- **步骤必须可执行**：测试步骤仅描述操作指令，验证内容写入预期结果
- **用例独立**：用例间不能存在依赖描述关系
- **格式固化**：禁止使用 Markdown 表格格式，全部使用 `**字段名**:` 换行格式

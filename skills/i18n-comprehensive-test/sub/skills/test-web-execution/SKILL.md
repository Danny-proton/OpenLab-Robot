# Test Web Execution

## 执行策略（最高优先级）

> **重要提醒：任务可能非常多且复杂，涉及大量用例的连续执行。以下规则必须严格遵守。**

### 1. 全局连续执行原则

- **用例之间默认连续执行，绝不主动暂停**
- 若用户未明确指示暂停，**整个测试流程必须一气呵成，连续执行直到全部用例完成**
- 即使任务量巨大、用例数量众多，也不得中途停止
- 多条用例执行时**必须串行执行，禁止并行**

### 2. 单用例内连续执行原则

- **单用例内多步骤之间也不得无故暂停**，按"执行 → 截图 → 验证 → 下一步"闭环推进
- 单个用例内的操作可能很复杂，包含多个步骤和多轮交互，**必须连续执行到该用例完成**
- 禁止在一个用例执行过程中插入无关操作或无故中断

### 3. 异常处理

- 出现用例信息缺失、操作连续失败、页面异常等情况，**必须退出当前用例执行**并在报告中说明原因
- 退出当前用例后，**继续执行后续用例**，不得影响其他用例的执行

## 浏览器操作工具

所有网页操作必须使用 **Chrome DevTools MCP 工具**。这些工具已作为标准工具直接暴露在你的工具列表中，**直接以工具调用方式使用**（如 `new_page`、`take_snapshot`、`click`），**不要**通过 `Bash` 执行 `npx` 命令或其他方式间接调用。

可用工具及用途：

| 工具                     | 用途                                              |
| ------------------------ | ------------------------------------------------- |
| `new_page`             | 打开新页签（禁止传 `isolatedContext`）          |
| `navigate_page`        | 在当前页签导航到 URL                              |
| `select_page`          | 切换到指定页签（必须设置 `bringToFront: true`） |
| `list_pages`           | 列出所有页签                                      |
| `close_page`           | 关闭页签                                          |
| `take_snapshot`        | 获取页面 a11y 树和元素 uid（操作前必调）          |
| `take_screenshot`      | 截图保存（必须指定 `filePath`）                 |
| `click`                | 点击元素（传入 `uid`）                          |
| `fill` / `fill_form` | 填写输入框                                        |
| `type_text`            | 键盘输入文本                                      |
| `press_key`            | 按键操作                                          |
| `hover`                | 鼠标悬停                                          |
| `drag`                 | 拖拽元素                                          |
| `upload_file`          | 文件上传                                          |
| `handle_dialog`        | 处理浏览器弹窗（alert/confirm/prompt）            |
| `evaluate_script`      | 执行 JS 代码                                      |
| `wait_for`             | 等待页面出现指定文本                              |

**禁止**使用其他浏览器自动化工具（Selenium / Playwright / Puppeteer 等）。

### 关键注意事项

- 操作前必须 `take_snapshot` 获取最新 uid（uid 每次操作后可能变化）
- 不同系统间导航时，用 `new_page` 打开新页签，不用 `navigate_page` 覆盖
- 点击某些按钮、表单提交后可能弹出新页签，在这些可能的操作后须使用 `list_pages` 确认，如弹出新页签，需切换至新页签，避免在新页签上操作旧页签

## 基本工作流

```
① 操作前 take_snapshot + take_screenshot 并行 → 定位元素 + 记录前状态
② 交互操作(click/fill/...)
③ 操作后 take_screenshot → 记录结果
④ 可能触发新页签的操作后，使用 list_pages 确认
```

- **操作前**：`take_snapshot` 获取 a11y 树和 uid，`take_screenshot` 记录页面状态，二者**并行发送**
- **操作后**：仅 `take_screenshot` 记录结果（必须指定 `filePath`）
- **禁止用 `take_snapshot` 的输出替代截图**
- 可能触发新页签的操作（点击登录按钮、表单提交等）后，使用 `list_pages` 查看是否有新页签

## 登录与会话管理（v1.1 新增 - Issue 2 & Issue 4）

> 本章节定义登录流程（含验证码处理）与会话失效后的重登修复流程。**根因说明**：原 skill 在会话失效后直接复用旧 uid 重填表单，但页面重载后 uid 已变更，导致"输入后无效"。本流程通过"重新快照 + 等待就绪 + 验证成功"三步修复。

### 1. 登录前置：读取凭证与环境

登录前**必须**从 `.memory/shared/environment.md` 读取：

- 系统入口 URL
- 用户名、密码
- 验证码策略（`captcha_strategy`：none / graphic / sms / unknown）
- 登录后预期跳转的 URL 或预期出现的登录态元素（用户菜单、头像、退出按钮等）

> **铁律**：凭证只从 memory 读取，**禁止再次询问用户**（用户已在 Phase 1 提供过）。

### 2. 登录流程（按验证码策略分支）

#### 策略 A：无验证码（captcha_strategy = none）

```
1. navigate_page 到登录 URL（或在已打开的登录页签上操作）
2. 等待页面 readyState=complete（用 evaluate_script 校验 document.readyState）
3. take_snapshot 获取登录表单最新 uid
4. fill 用户名输入框（uid）
5. fill 密码输入框（uid）
6. take_screenshot 记录填值后状态
7. click 登录按钮（uid）
8. 等待跳转（wait_for 等待登录后预期的页面文本出现，或 sleep 2-3s）
9. 验证登录成功（见下文"登录成功验证"）
```

#### 策略 B：图形/滑块验证码（captcha_strategy = graphic）

```
1-6. 同策略 A（填用户名、密码）
7. take_snapshot 检测验证码元素（img.captcha、.slider-captcha、canvas 等）
8. 若检测到验证码：
   a. take_screenshot 保存当前验证码画面到 ./login_captcha_<timestamp>.png
   b. 通过 AskUserQuestion 通知用户：
      - question: "检测到登录验证码。请在浏览器中手动完成验证码，完成后回复'已完成'。"
      - 选项: ["已完成验证码", "跳过登录（标记用例为 BLOCKED）"]
   c. 用户确认"已完成"后，take_snapshot 重新获取登录按钮 uid（验证码操作可能改变 DOM）
   d. click 登录按钮
9. 若未检测到验证码（误判为 graphic 但实际无）：按策略 A 步骤 7-9 执行
10. 验证登录成功
```

#### 策略 C：短信/邮件验证码（captcha_strategy = sms）

```
1-6. 同策略 A（填用户名、密码）
7. click "登录"或"发送验证码"按钮（视页面设计）
8. 通过 AskUserQuestion 请求用户输入收到的验证码：
   - question: "已触发短信/邮件验证码发送。请输入收到的验证码。"
   - 输入类型: 文本
9. 等待用户输入（用户回复中包含验证码）
10. take_snapshot 获取验证码输入框 uid
11. fill 验证码（uid）
12. click 登录/确认按钮
13. 验证登录成功
```

#### 策略 D：不确定（captcha_strategy = unknown）

```
1. 按策略 A 执行到步骤 7（click 登录）
2. sleep 2s 后 take_snapshot + 当前 URL 检查
3. 若仍停留在登录页 或 检测到验证码元素：
   → 切换为策略 B（图形验证码）或策略 C（短信验证码），按页面实际元素判断
4. 若已跳转离开登录页 → 验证登录成功
```

### 3. 登录成功验证（必做）

登录操作后，**必须**通过以下任一方式确认登录成功，否则视为失败：

1. **URL 校验**：当前 URL 不再是登录页 URL（且不含 `/login`、`/signin` 等路径片段）
2. **登录态元素校验**：`take_snapshot` 中存在用户菜单、头像、退出/注销按钮、欢迎语等登录态元素
3. **页面文本校验**：`wait_for` 等待登录后首页的特征文本出现

若验证失败：
- 截图保存失败现场
- 检查是否有错误提示（账号密码错误、验证码错误、账号锁定）
- 若是验证码错误 → 回到对应策略的验证码处理步骤重试一次
- 若是账号密码错误 → **不要重试**，标记用例 BLOCKED 并报告（凭证来自 memory，重试无意义）
- 若账号锁定 → 标记 BLOCKED，通过 AskUserQuestion 通知用户

### 4. 会话失效检测

执行用例过程中，**每次操作前/后**检测会话是否失效：

```
会话失效信号（满足任一即判定失效）:
- 当前 URL 跳转回登录页（含 /login、/signin、/auth）
- take_snapshot 中出现登录表单（username/password input）
- 页面出现"登录已过期"、"会话超时"、"请重新登录"等文本
- evaluate_script 检查特定登录态元素不存在
- 接口返回 401/403（若可观测）
```

### 5. 会话失效与重登修复流程（Issue 4 核心修复）

检测到会话失效后，**立即**按以下流程修复（不打断用例整体执行）：

```
STEP 1: 标记中断点
  - 记录当前用例编号、当前步骤、当前目标页面 URL
  - 截图保存会话失效现场 → ./<用例编号>/session_expired_<timestamp>.png

STEP 2: 重读凭证（禁止询问用户）
  - 从 .memory/shared/environment.md 重新读取 username、password、captcha_strategy
  - 若文件读取失败 → 标记用例 BLOCKED 并报告"凭证读取失败"

STEP 3: 重新快照（修复"输入后无效"根因）
  - 关键：旧 uid 在页面重载/跳转后已失效，必须重新 take_snapshot
  - 在当前登录页上 take_snapshot，获取全新的用户名/密码输入框 uid
  - 若当前不在登录页（被重定向到其他页），navigate_page 回登录 URL 后再 take_snapshot

STEP 4: 等待页面就绪
  - evaluate_script 校验 document.readyState === 'complete'
  - 校验用户名输入框可交互（visible & enabled）
  - 未就绪则 sleep 1s 重试，最多 3 次；仍不就绪 → BLOCKED

STEP 5: 重新登录
  - 按第 2 节的对应验证码策略执行登录流程
  - 验证码策略为 graphic/sms 时，按策略 B/C 暂停等待人工
  - 登录后按第 3 节验证成功

STEP 6: 恢复执行
  - 登录成功后，navigate_page 回 STEP 1 记录的目标页面 URL
  - take_snapshot 重新获取页面 uid
  - 从中断的步骤继续执行用例
  - 在用例报告中记录"发生会话失效并成功重登"事件

STEP 7: 重登失败处理
  - 若单次重登失败（验证码错误除外），最多重试 1 次
  - 第二次仍失败 → 标记用例 BLOCKED，原因记为"会话失效后重登失败"
  - 继续执行后续用例（不阻塞整体流程）
```

### 6. 登录态保持规则

- **禁止关闭已登录页签**：登录成功后的页签必须保持打开，后续用例通过 `select_page` 切换
- **禁止覆盖登录页签**：访问新系统/新页面时用 `new_page` 新建页签
- **长时停顿后主动校验**：若用例间停顿超过 5 分钟，下一个用例执行前主动校验会话状态

## 测试记忆库

执行测试用例前及执行过程中，必须优先参考项目测试记忆库 `.memory/` 中的内容。记忆库中沉淀了系统结构、页面元素、操作流程、已知问题等经验，能显著提升执行效率和准确性。

### 记忆库读取流程

#### 1. 冷启动：执行前必读

在开始执行任何测试用例前，依次读取以下内容：

| 文件                                  | 用途                             | 读取时机   |
| ------------------------------------- | -------------------------------- | ---------- |
| `.memory/README.md`                 | 了解系统概览和目录结构           | 首次执行时 |
| `.memory/shared/environment.md`     | 获取当前环境、域名映射、登录状态 | 每次执行前 |
| `.memory/shared/execution_rules.md` | 获取通用导航、页面模式、注意事项 | 每次执行前 |

#### 2. 按用例定位模块记忆

根据用例中涉及的系统、菜单、页面、业务对象，定位 `.memory/modules/<module-name>/`，按顺序读取：

| 文件                | 用途                                         | 读取时机         |
| ------------------- | -------------------------------------------- | ---------------- |
| `README.md`       | 了解模块用途、入口 URL、导航结构             | 首次接触该模块时 |
| `objects.md`      | 了解页面对象、字段、按钮、弹窗、关键业务对象 | 首次接触该模块时 |
| `guide.md`        | 了解稳定执行流程                             | 遇到流程不清晰时 |
| `known_issues.md` | 了解已知问题和规避方案                       | 执行失败或异常时 |
| `cases/INDEX.md`  | 查看历史执行经验                             | 遇到类似场景时   |

#### 3. 跨模块流程

如果用例涉及多个模块的交互操作（如跨系统跳转、跨域名导航），额外读取：

- `.memory/shared/cross_module_flows.md`（如果存在）

### 记忆优先原则

执行过程中遇到问题时，按以下优先级依次尝试：

1. **当前用例上下文** — 优先使用用例中已有的信息和步骤
2. **模块记忆库** — 查阅对应模块的 `objects.md`、`guide.md`、`known_issues.md`
3. **共享规则** — 查阅 `.memory/shared/execution_rules.md`、`.memory/shared/environment.md`
4. **用户询问** — 只有记忆库中无相关信息时，才向用户提问

> **关键要求**：在点击按钮、填写表单、处理弹窗、导航跳转等操作前，**先检查记忆中是否有相关经验**。遇到元素找不到、点击无响应、页面不匹配等问题时，**立即查阅** `known_issues.md` 和 `objects.md`，不要盲目重试。

### 经验沉淀

执行完成后，将新发现的稳定经验沉淀回记忆库：

- 当前模块特有经验 → 写入 `.memory/modules/<module>/`
- 跨模块通用经验 → 写入 `.memory/shared/`
- 可跨项目复用的 Web 操作经验 → 沉淀到 test-execution skill

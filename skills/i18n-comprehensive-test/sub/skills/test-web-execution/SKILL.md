# Test Web Execution

## 执行策略（最高优先级）

> **重要提醒：任务可能非常多且复杂，涉及大量用例的连续执行。以下规则必须严格遵守。**

### 1. 全局连续执行原则

- **用例之间默认连续执行，绝不主动暂停**
- 若用户未明确指示暂停，**整个测试流程必须一气呵成，连续执行直到全部用例完成**
- 即使任务量巨大、用例数量众多，也不得中途停止
- 多条用例执行时**必须串行执行，禁止并行**

### 2. 单用例内连续执行原则

- **单用例内多步骤之间也不得无故暂停**，按"执行 → 截图 → 验证 → 下一步"闭环推进
- 单个用例内的操作可能很复杂，包含多个步骤和多轮交互，**必须连续执行到该用例完成**
- 禁止在一个用例执行过程中插入无关操作或无故中断

### 3. 异常处理

- 出现用例信息缺失、操作连续失败、页面异常等情况，**必须退出当前用例执行**并在报告中说明原因
- 退出当前用例后，**继续执行后续用例**，不得影响其他用例的执行

## 浏览器操作工具

所有网页操作必须使用 **Chrome DevTools MCP 工具**。这些工具已作为标准工具直接暴露在你的工具列表中，**直接以工具调用方式使用**（如 `new_page`、`take_snapshot`、`click`），**不要**通过 `Bash` 执行 `npx` 命令或其他方式间接调用。

可用工具及用途：

| 工具                     | 用途                                              |
| ------------------------ | ------------------------------------------------- |
| `new_page`             | 打开新页签（禁止传 `isolatedContext`）          |
| `navigate_page`        | 在当前页签导航到 URL                              |
| `select_page`          | 切换到指定页签（必须设置 `bringToFront: true`） |
| `list_pages`           | 列出所有页签                                      |
| `close_page`           | 关闭页签                                          |
| `take_snapshot`        | 获取页面 a11y 树和元素 uid（操作前必调）          |
| `take_screenshot`      | 截图保存（必须指定 `filePath`）                 |
| `click`                | 点击元素（传入 `uid`）                          |
| `fill` / `fill_form` | 填写输入框                                        |
| `type_text`            | 键盘输入文本                                      |
| `press_key`            | 按键操作                                          |
| `hover`                | 鼠标悬停                                          |
| `drag`                 | 拖拽元素                                          |
| `upload_file`          | 文件上传                                          |
| `handle_dialog`        | 处理浏览器弹窗（alert/confirm/prompt）            |
| `evaluate_script`      | 执行 JS 代码                                      |
| `wait_for`             | 等待页面出现指定文本                              |

**禁止**使用其他浏览器自动化工具（Selenium / Playwright / Puppeteer 等）。

### 关键注意事项

- 操作前必须 `take_snapshot` 获取最新 uid（uid 每次操作后可能变化）
- 不同系统间导航时，用 `new_page` 打开新页签，不用 `navigate_page` 覆盖
- 点击某些按钮、表单提交后可能弹出新页签，在这些可能的操作后须使用 `list_pages` 确认，如弹出新页签，需切换至新页签，避免在新页签上操作旧页签

## 基本工作流

```
① 操作前 take_snapshot + take_screenshot 并行 → 定位元素 + 记录前状态
② 交互操作(click/fill/...)
③ 操作后 take_screenshot → 记录结果
④ 可能触发新页签的操作后，使用 list_pages 确认
```

- **操作前**：`take_snapshot` 获取 a11y 树和 uid，`take_screenshot` 记录页面状态，二者**并行发送**
- **操作后**：仅 `take_screenshot` 记录结果（必须指定 `filePath`）
- **禁止用 `take_snapshot` 的输出替代截图**
- 可能触发新页签的操作（点击登录按钮、表单提交等）后，使用 `list_pages` 查看是否有新页签

## 测试记忆库

执行测试用例前及执行过程中，必须优先参考项目测试记忆库 `.memory/` 中的内容。记忆库中沉淀了系统结构、页面元素、操作流程、已知问题等经验，能显著提升执行效率和准确性。

### 记忆库读取流程

#### 1. 冷启动：执行前必读

在开始执行任何测试用例前，依次读取以下内容：

| 文件                                  | 用途                             | 读取时机   |
| ------------------------------------- | -------------------------------- | ---------- |
| `.memory/README.md`                 | 了解系统概览和目录结构           | 首次执行时 |
| `.memory/shared/environment.md`     | 获取当前环境、域名映射、登录状态 | 每次执行前 |
| `.memory/shared/execution_rules.md` | 获取通用导航、页面模式、注意事项 | 每次执行前 |

#### 2. 按用例定位模块记忆

根据用例中涉及的系统、菜单、页面、业务对象，定位 `.memory/modules/<module-name>/`，按顺序读取：

| 文件                | 用途                                         | 读取时机         |
| ------------------- | -------------------------------------------- | ---------------- |
| `README.md`       | 了解模块用途、入口 URL、导航结构             | 首次接触该模块时 |
| `objects.md`      | 了解页面对象、字段、按钮、弹窗、关键业务对象 | 首次接触该模块时 |
| `guide.md`        | 了解稳定执行流程                             | 遇到流程不清晰时 |
| `known_issues.md` | 了解已知问题和规避方案                       | 执行失败或异常时 |
| `cases/INDEX.md`  | 查看历史执行经验                             | 遇到类似场景时   |

#### 3. 跨模块流程

如果用例涉及多个模块的交互操作（如跨系统跳转、跨域名导航），额外读取：

- `.memory/shared/cross_module_flows.md`（如果存在）

### 记忆优先原则

执行过程中遇到问题时，按以下优先级依次尝试：

1. **当前用例上下文** — 优先使用用例中已有的信息和步骤
2. **模块记忆库** — 查阅对应模块的 `objects.md`、`guide.md`、`known_issues.md`
3. **共享规则** — 查阅 `.memory/shared/execution_rules.md`、`.memory/shared/environment.md`
4. **用户询问** — 只有记忆库中无相关信息时，才向用户提问

> **关键要求**：在点击按钮、填写表单、处理弹窗、导航跳转等操作前，**先检查记忆中是否有相关经验**。遇到元素找不到、点击无响应、页面不匹配等问题时，**立即查阅** `known_issues.md` 和 `objects.md`，不要盲目重试。

### 经验沉淀

执行完成后，将新发现的稳定经验沉淀回记忆库：

- 当前模块特有经验 → 写入 `.memory/modules/<module>/`
- 跨模块通用经验 → 写入 `.memory/shared/`
- 可跨项目复用的 Web 操作经验 → 沉淀到 test-execution skill

## 截图规范

每个操作步骤执行后**必须立即截图**，不得跳步、合并或省略。

- 必须指定 `filePath` 保存到本地
- 路径格式：`<用例编号>/step_<序号>_<操作描述>.png`
- `evaluate_script` JS 操作也必须截图
- 禁止连续执行多个操作再截图

## 输入处理

1. **自然语言指令**：直接解析并执行
2. **Excel 测试用例**：先用 `xlsx-to-markdown` 转为 Markdown

### Excel 用例处理流程

```bash
python "${CLAUDE_SKILL_DIR}/xlsx-to-markdown/xlsx_to_markdown.py" --input "<Excel文件路径>"
```

读取生成的 `<Excel文件名>.md`，按用例执行。**禁止**使用其他方式读取 Excel。

## 任务管理

为每个用例创建独立任务，将用例完整详情带入 `description`。

**subject**：`TC-序号: 用例名称（简写）`

**description**：

```
用例编号: <CaseNumber>
用例名称: <CaseName>
用例描述: <Description>
预置条件: <Prerequisites>
测试步骤: <Test procedure>
预期结果: <Expected result>
```

---

## 用例编号与目录规则（最高优先级）

1. 用例中明确包含编号的，**直接使用**，如 `FRS07_CSZSQ_01_002_001`
2. **禁止自行编造或修改用例编号**（添加前缀后缀、替换字符、截断、合并）
3. 没有明确编号时，使用 `TC-序号` 作为编号
4. 截图和报告文件保存至 `<用例编号>/` 目录下

---

## 调试模式

若输入不是以用例 Excel 启动的正式测试任务（即用户通过自然语言发起的浏览、抓取数据、点击操作等），视为**调试模式**。

- **调试模式下无须保存截图**
- **无须输出测试报告**
- 按需操作即可，灵活执行

---

## 输出要求

正式用例测试完成后输出结构化报告，并保存到 `<用例编号>/test_report.md`。

模板和规则见 `test-report` 子技能。

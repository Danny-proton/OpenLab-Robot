# 测试设计 PRD — 详细设计

## 目标

从待测 Agent 的 PRD/SPEC + 代码扫描记忆，系统化生成高质量测试用例。

## 输入

| 输入 | 格式 | 说明 |
|------|------|------|
| Agent PRD | 文本/Markdown | 产品需求文档，描述 Agent 的功能和约束 |
| Agent SPEC | 文本/Markdown | 技术规格，描述工具/接口/数据结构 |
| 代码扫描记忆 | JSON | 自动扫描 Agent 代码提取的：工具列表/参数 schema/Advisor 链/Memory 配置 |
| 历史 case | YAML | 已有用例（迭代增强时） |
| 历史错误 | JSON | 上一轮测试的 F1-F8 错误分布（迭代时） |

## 流程

### 阶段 1: 需求分解

**输入**：Agent PRD
**输出**：功能点列表 + 非功能点列表 + 约束列表 + 边界列表

```
PRD 文本
  ↓
1. 功能点提取（"Agent 能做什么"）
   - 账户查询、转账、理财购买...
  ↓
2. 非功能点提取（"Agent 做得好不好"）
   - 响应时间 < 3s、支持 1000 并发...
  ↓
3. 约束提取（"Agent 不能做什么"）
   - 禁止直接批准、必须身份验证...
  ↓
4. 边界提取（"Agent 的极限在哪"）
   - 最大转账金额、最长对话轮数...
```

### 阶段 2: SPEC 解析

**输入**：Agent SPEC + 代码扫描记忆
**输出**：可测试断言列表

```
SPEC + 代码扫描
  ↓
1. 工具列表提取（Agent 有哪些 @Tool）
   - 工具名、参数 schema、返回格式
  ↓
2. Advisor 链提取（Agent 有哪些前置/后置检查）
   - InputValidationAdvisor、FallbackAdvisor...
  ↓
3. 业务规则提取（SPEC 里的 if-then 逻辑）
   - "如果流水波动 > 30% 则标记高风险"
  ↓
4. 可测试断言生成
   - "调用 queryBalance 后必须返回数字"
   - "调用 transferMoney 前必须调用 verifyIdentity"
   - "流水波动 > 30% 时 final_answer 必须含'风险'"
```

### 阶段 3: 测试因子提取

**输入**：需求分解 + SPEC 解析
**输出**：测试因子矩阵

**方法**：

| 方法 | 适用 | 示例 |
|------|------|------|
| 等价类划分 | 输入参数 | 金额：0 / 1-9999 / 10000-99999 / >100000 |
| 边界值分析 | 数值参数 | 金额边界：0, 1, 9999, 10000, 99999, 100000 |
| 状态迁移 | 多轮对话 | 空闲→查询中→确认中→执行中→完成 |
| 正交实验 | 多参数组合 | 金额 × 期限 × 风险等级 的正交表 |
| 决策表 | if-then 规则 | 评分<650 AND 负债>70% → 拒件 |
| 场景法 | 业务流程 | 正常转账全流程 / 异常中断恢复 |

### 阶段 4: 测试方法库选择

**输入**：测试因子
**输出**：选定的测试方法

```yaml
# test_method_library.yaml
methods:
  - id: eq-class
    name: 等价类划分
    applicable_factors: [input_param, range]
    description: 把输入域分成等价类，每类选代表值

  - id: boundary
    name: 边界值分析
    applicable_factors: [numeric, range]
    description: 测试边界值和边界附近的值

  - id: state-transition
    name: 状态迁移
    applicable_factors: [multi_turn, state]
    description: 测试状态转换路径

  - id: orthogonal
    name: 正交实验
    applicable_factors: [multi_param, combination]
    description: 用正交表减少组合数

  - id: decision-table
    name: 决策表
    applicable_factors: [rule, if_then]
    description: 测试所有规则组合

  - id: scenario
    name: 场景法
    applicable_factors: [workflow, business_process]
    description: 测试完整业务流程
```

### 阶段 5: 用例生成

**输入**：测试因子 × 测试方法
**输出**：agent-eval 格式 case YAML

Agent 通过子 skill 自己生成，不在脚本里调 LLM。

### 阶段 6: 格式化检查

**输入**：生成的 case YAML
**输出**：通过/不通过 + 错误列表

检查项：
- [ ] YAML 格式正确
- [ ] 必填字段齐全（id/name/input/expected）
- [ ] id 唯一
- [ ] expected.contains 非空
- [ ] expected_tools 格式正确

### 阶段 7: 用例自检

**输入**：通过格式化检查的 case
**输出**：质量评分 + 改进建议

**检查维度**：

| 维度 | 检查项 | 权重 |
|------|--------|------|
| SPEC 完整性 | 是否覆盖了 PRD 的所有功能点 | 0.15 |
| 用例完整性 | 是否有前置条件/步骤/预期/断言 | 0.10 |
| 功能点覆盖度 | 10 维度是否都有用例 | 0.15 |
| DFX 覆盖度 | 性能/安全/兼容/可靠/韧性的覆盖 | 0.15 |
| 有效用例率 | 无过时/重复/不可执行用例 | 0.10 |
| 执行可行性 | 用例是否能实际跑通 | 0.10 |
| 无二义性 | 步骤/预期是否清晰无歧义 | 0.10 |
| 长度合理 | 用例不过长（< 10 步）| 0.05 |
| 断言可验证 | expected 是否可自动验证 | 0.10 |

### 阶段 8: 用例自优化（迭代增强）

**触发**：完成一轮测试后

**流程**：
```
1. 读取上一轮 diagnoser 结果（F1-F8 分布）
2. 分析错误集中类型
3. 识别 spec 缺口
4. 识别用例质量问题
5. 生成增强建议
6. 与人确认
7. 更新 cases YAML
```

**增强策略**：

| 错误集中类型 | 增强策略 |
|-------------|---------|
| F1-F2 集中 | 补充任务类型识别用例，增强 PRD 解析 |
| F3 集中 | 增加工具选择边界用例，细化 expected_tools |
| F4 集中 | 增加参数边界用例，细化 arguments 断言 |
| F5 集中 | 增加异常恢复流程用例 |
| F7 集中 | 细化 expected.contains 关键词 |
| F8 集中 | 调整 expected_steps，增加效率用例 |
| 某维度 0 用例 | 补充该维度用例 |
| 某维度 100% 通过 | 增加边界/异常用例 |

## 与 Agent 的交互

### 需要和人确认的环节

1. **需求分解后**：确认功能点/约束/边界是否完整
2. **用例生成后**：确认用例是否符合业务理解
3. **用例自检后**：确认质量问题是否需要修复
4. **用例自优化后**：确认增强建议是否接受

### 自动化的环节

1. SPEC 解析（代码扫描自动提取）
2. 测试因子提取（方法库自动匹配）
3. 格式化检查（脚本自动检查）
4. 用例自检（脚本计算质量评分）
5. 错误分布分析（脚本自动统计）

## 用例质量衡量

### 维度

| 维度 | 指标 | 衡量方法 |
|------|------|---------|
| 覆盖度 | 功能点覆盖率 | 已覆盖功能点 / 总功能点 |
| 覆盖度 | 维度覆盖率 | 有用例的维度 / 10 |
| 覆盖度 | DFX 覆盖率 | 已覆盖 DFX / 7 |
| 质量 | 二义性率 | 有二义性的用例 / 总用例 |
| 质量 | 有效用例率 | 可执行用例 / 总用例 |
| 质量 | 平均步数 | 所有用例 steps 数的均值 |
| 迭代 | 新增用例数 | 本轮新增 / 总用例 |
| 迭代 | 废弃用例数 | 本轮废弃 / 总用例 |
| 迭代 | 质量分变化 | 本轮质量分 - 上轮质量分 |

## 〇、文档说明

### 0.1 本文档写什么

本文档描述 OpenLab Robot 客户端中「新建测试任务 → Agent 评测」页签的**完整用户旅程**，逐页给出高保真文字描述：每页回答"这页给用户看什么、用户在这页做什么、点完去哪"，包括布局分区、元素内容、交互动效、状态变化、页面流转。**不给出具体实现代码**。

### 0.2 与既有产物的关系

- **入口框架复用**：Agent 评测与「批量测试执行」共用同一套入口框架——「新建会话」下「新建测试任务」按钮、三个页签（测试设计 / 测试执行 / Agent 评测）、同名文件夹工作区。本旅程启用此前置灰的「Agent 评测」页签；页签内的导航采用**左侧一级菜单 + 可唤起的流程栏**（见 1.2），各环节可独立建设、独立导出。
- **底层能力调用子 skill**：页面上的一切专业动作（指标计算、用例生成、诊断、优化）都不是网页自己实现，而是调用 `skills/agent-eval-v1.1` 的子 skill 与脚本。每页描述中会标注其调用的底层能力。
- **业界对标**：整体工作流对齐 Confident AI（deepeval）的评测范式——指标（Metrics）→ 数据集（Datasets / Goldens）→ 实验执行（Experiments）→ 报告洞察（Insights）→ 回归与优化（A/B Regression），并在此基础上长出两块差异化能力：**三层级系统适配（黑盒/半白盒/白盒）** 与 **用例自优化双闭环**。

### 0.3 演示设计思路（价值点 + 用户旅程线 + 解决方案面）

本文档按"以点带面"组织，五个环节既是一条完整旅程线，也各自是一个可单独演示的价值点：

| 环节 | 用户旅程线（流程框架演示） | 单点价值（单点展开演示） |
|------|--------------------------|--------------------------|
| 1 评测规则指标 | 回答"拿什么评" | **评测体系演示**：从哪些维度诊断和优化，直观得到业务预期 |
| 2 数据集创建 | 回答"用什么数据评" | **用例生成演示**：生成了哪些用例、针对哪些诊断目标而使用 |
| 3 评估 Agent 执行 | 回答"怎么评" | **诊断能力演示**：能诊断出什么问题、可用的优化策略是什么 |
| 4 报告可视化 | 回答"评得怎么样" | **指标透视演示**：从一个指标打开，看到用例、分数、根因、建议一整串 |
| 5 优化 | 回答"怎么变得更好" | **优化执行演示**：自动优化 + 用户交互解决问题的能力 |

## 一、全局框架（所有页共享）

### 1.1 入口

用户在「新建会话」下方点击「新建测试任务」，弹出三个页签：测试设计、测试执行、Agent 评测。**本旅程中「Agent 评测」页签由置灰变为可用**， hover 时有微光浮动效，点击进入。首次进入时任务单独新建一个与项目同名的文件夹作为工作区；历史评测任务在 Agent 评测界面左侧一级菜单的「执行历史」页签中查看，点击可回看任意历史任务的报告与产物。

### 1.2 两种使用模式：独立建设（默认）与流程引导

Agent 评测支持两种使用模式，**默认进入独立建设模式**，流程栏不是一开始就唤起的。

**模式一 · 独立建设模式（默认）**

进入任务后，界面**左侧是一级菜单（竖向页签）**，依次为五个大环节页签 + 一个执行历史页签：

**评测规则指标｜数据集｜评估执行｜报告｜优化｜执行历史**

- 每个大环节页签都是**独立的进入方式**：用户可以从任意一个环节单独进入建设，不必走完整流程。例如只建一套评测指标、只造一批数据集；
- 各环节独立建设的产物都可**单独导出**（见 1.4），落工作区、可复用：比如在"评测规则指标"里建好评测集后导出，在"数据集"里建设数据集后导出；后续别的评测任务（或流程模式）可以直接引用这些已建好的资产；
- 各环节之间通过工作区资产互相衔接而非强制顺序：如数据集建设页可引用已存在的评测集，评估执行页可自选"评测集 × 数据集"组合；
- 页签选中态有滑动高亮动效；已建有资产的环节页签上显示资产角标（如"评测集 ×2""数据集 v1.1"）。

**模式二 · 流程引导模式（点"开始"唤起）**

一级菜单**最下方有一个醒目的「开始评测流程」主按钮**。点击后：

- 左侧一级菜单被**五环节流程栏覆盖替换**（菜单向左滑出、流程栏自左滑入的覆盖动效），进入 ①→⑤ 的顺序引导；
- 流程栏自上而下展示五个环节卡片：**① 评测规则指标 → ② 数据集 → ③ 评估执行 → ④ 报告 → ⑤ 优化**。当前环节卡片放大、高亮描边、带呼吸光效；已完成环节左上角绿色对勾，点击可回看（重新编辑会提示"将影响后续环节产物"）；未开始环节灰色半透明不可点击；环节间连接线随完成进度逐段点亮，有流动光点动效；
- 流程模式中若用户此前在独立模式下已建好对应资产（如已有评测集、数据集），进入该环节时默认选中最新资产，可直接"沿用并跳过"或重新编辑；
- **流程栏可收起**：点击流程栏顶部的收起按钮，流程栏向左收起、一级菜单滑回，任务回到独立建设模式；**进行中的流程不丢失**——在「执行历史」页签中该任务显示"流程进行中 · 停在环节X"的进度徽标，点击即可重新唤起流程栏并恢复到离开时的环节。

其中「评估执行」环节是复合环节：内部含**适配自检**与**批量执行**两个子环节；批量执行时流程栏内该环节按批次数动态裂变出多个子卡片（一个变多个有展开动效），与《测试执行任务PRD.md》中"任务执行过程 × N 个批次"的表现一致。

### 1.3 页面骨架（每页通用）

- **左侧**：一级菜单（独立建设模式）或五环节流程栏（流程引导模式，可收起），两者占据同一栏位、互相切换；
- **中区**：当前环节的主操作区，内容以卡片式分区块呈现；
- **底部**：环节级主操作按钮，随模式变化——独立建设模式下是「保存」与「导出」；流程引导模式下是「确认，进入下一环节」（点击后流程栏中当前环节打勾、下一环节点亮的过场动效）；
- **顶部**：项目名（默认按规则自动生成、可编辑）+ 当前环节名 + 当前模式标识（独立建设 / 流程引导）+ 工作区路径提示。

### 1.4 各环节独立建设的产物与导出

| 环节页签 | 独立建设产物 | 导出形式 |
|----------|-------------|----------|
| 评测规则指标 | 评测集（指标组合 + 权重 + 硬失败门槛） | 评测集文件（YAML/JSON），可导入复用 |
| 数据集 | 带版本号的数据集（题目 + GT + 期望工具链 + 断言） | 数据集文件（Excel/YAML/JSON），可导入复用 |
| 评估执行 | 执行记录与 trace、诊断结论 | 执行结果包（trace + 归因清单） |
| 报告 | 各类报告资产 | MD / HTML / PDF 报告、指标透视链快照 |
| 优化 | patch 记录、优化后的评测集/数据集新版本 | patch diff 包、迭代报告 |
| 执行历史 | 历史任务与进行中流程的入口 | 归档包（含全部产物） |

# 第一部分：评测规则指标创建

> 本环节回答"**拿什么标准评这个 Agent**"。最终产物是一份**本次评测的指标组合（评测集）**：预置指标勾选结果 + 自定义业务规则 + 各指标权重 + 硬失败门槛。
> 对应演示点：**评测体系演示**——支持从哪些维度诊断和优化，直观得到通过本方案可以达到的业务预期。

## 页面 1.1 指标中心（预置规则库首页）

**页面价值**：让用户一眼看到"评测体系的全貌"——平台预置了哪些评测规则，覆盖哪些维度，做到"全面 + 重点突出"，而不是全量发散。

**布局与元素**：

- 顶部是环节标题「评测规则指标创建」和一句话引导："选择本次评测要用的规则指标，也可以添加你自己的业务规则"。
- 主体按**指标族分组**展示预置指标卡片，每组一个色带标题，共六组：
  1. **硬指标（代码可判定）**——5 张卡：任务成功 `task_success`、工具调用正确性 `tool_correctness`、业务规则覆盖 `business_rule_coverage`、输出结构合规 `output_schema_validity`、效率 `efficiency`。卡片右上角带"规则判定"徽章（表示像 deepeval 的代码评估一样由确定性公式判定，不依赖大模型）。
  2. **软指标（AI 评判）**——3 张卡：答案相关性 `answer_relevance`、证据忠实度 `evidence_faithfulness`、步骤效率 `step_efficiency`。卡片右上角带"G-Eval"徽章（表示由 LLM-as-a-Judge 按评分标准打分，0 / 0.5 / 1 三档）。
  3. **业务价值族 `bv_*`**——4 张卡：客户体验、业务决策准确率、SOP 完整性、术语准确性。带"人工标注校准"徽章。
  4. **性能效率族 `pe_*`**——2 张卡：Token 消耗、工具效率。
  5. **风险控制族 `rc_*`**——1 张卡：风险分类准确性。
  6. **评审团视角（9 Judge）**——9 张卡：领域、工具轨迹、工作流、忠实度、回归、安全 6 个规则型 Judge + 守门人、优化规划师、补丁撰写者 3 个决策型 Judge。这组卡片样式区别于前五组（评审团成员头像式图标），强调"多视角评审"而非"打分公式"。
- 每张指标卡片包含：指标中文名、一句话说明、判定方式徽章、默认权重、一个勾选框。勾选时卡片翻面动效，背面显示该指标"能诊断出什么问题"（例如工具调用正确性 → "能发现 F3 工具选择失败、F4 工具参数失败"）。

**交互与动效**：

- 鼠标悬停卡片时卡片浮起并展开完整描述；
- 页面顶部有搜索框和筛选器（按判定方式：规则判定 / AI 评判 / 人工标注）；
- 左下角实时统计："已选 X 项指标 · 覆盖 Y 个诊断维度"，随勾选即时更新并伴随数字滚动动效；
- 「+ 新建业务规则」按钮固定在指标中心右上角，主色描边，引导用户进入页面 1.3。

**调用的底层能力**：读取 `examples/.agent-eval/metrics/*.yaml` 指标定义文件与 guide 03 指标合约渲染卡片。

**出口**：勾选完成后点击底部「下一步：配置指标组合」进入页面 1.4；或先点「+ 新建业务规则」进入页面 1.3。

## 页面 1.2 指标详情抽屉（子页）

**页面价值**：让用户在勾选前看懂一个指标的判定逻辑，建立信任感。

**布局与元素**：从右侧滑出的抽屉（左侧一级菜单/流程栏保持可见），内容分四块：

1. **定义**：该指标衡量什么、取值范围（如 0/1、0-5 分制归一）、聚合公式（如 `0.5 × 工具召回 + 0.3 × 顺序分 + 0.2 × 禁用工具分`）；
2. **判定方式**：规则判定类展示判定组件清单（如"必调工具是否全部命中、禁用工具是否被调用、期望顺序的最长公共子序列"）；AI 评判类展示评判标准文本与三档评分说明；人工标注类说明标注入口在"数据集创建"环节；
3. **能诊断什么**：该指标低分映射到的失败类型（F1-F8）与典型业务含义；
4. **示例**：一条虚拟用例在该指标下的得分演算过程（逐组件打勾/打叉，最后聚合成总分）。

**交互与动效**：抽屉滑入滑出流畅；公式中的组件名可点击跳转到对应解释；底部按钮「在指标组合中启用/停用」直接切换勾选状态，与页面 1.1 的勾选双向同步。

**出口**：关闭抽屉回到指标中心。

## 页面 1.3 自定义业务规则创建/编辑页

**页面价值**：把用户脑子里的业务 know-how（"这个场景必须走人工复核""金额超过 5 万必须给风险提示"）落成可执行的评测规则。这是"手动编辑添加业务规则"的入口。

**布局与元素**：全屏表单页，分四个区：

1. **规则基本信息**：规则名称（必填）、业务场景描述、适用条件（作用于哪些类型的用例）。
2. **判定方式**（三选一的单选卡片，选中卡片展开对应配置项）：
   - **代码断言**（对标 deepeval 的规则评估）：可叠加多条断言——关键词包含、正则匹配、JSON 字段数值比较、工具必须/禁止调用、调用顺序约束。每条断言一行，可增删；
   - **AI 评判**（对标 G-Eval）：填写"评分标准"自然语言描述 + 可选"评分步骤"（先检查什么、再检查什么），系统按此让 LLM judge 给出 0 / 0.5 / 1 三档分；下方有"用一条示例用例试评"按钮，点击后即时展示一次评判结果与理由，帮助用户校准评分标准的措辞；
   - **人工标注**：该规则的得分来自数据集环节的人工标注，仅需要定义 1-5 分的评分锚点说明。
3. **计分设置**：权重（默认 0.10，滑条调节）、是否为硬失败规则（开启后命中即整条用例判失败，如"禁用工具被调用"）、及格阈值。
4. **AI 辅助起草**（可选区）：一个输入框"用一句话描述你的规则"，提交后由底层子 skill 生成规则草案（判定方式建议 + 断言/评分标准初稿），用户在此基础上修改。草案区域与手工表单区左右并排，方便对照采纳。

**交互与动效**：判定方式切换时三个配置区有手风琴展开动效；"试评"按钮点击后有加载动画，结果以聊天气泡样式滑出；保存时表单校验（必填项、权重范围、正则合法性）不通过处红框抖动提示。

**调用的底层能力**：保存时生成一份新的指标 YAML（同 `metrics/*.yaml` 合约）写入工作区；"AI 辅助起草"与"试评"调用 `test-case-generator` 子 skill 的生成能力与 LLM judge。

**出口**：「保存并返回指标中心」——新规则以"自定义"徽章卡片出现在指标中心对应分组末尾，默认已勾选。

## 页面 1.4 指标组合（评测集）配置页

**页面价值**：把选中的指标组装成一份可复用的"评测集"，明确各指标在总分中的话语权。

**布局与元素**：

- 左侧：**已选指标清单**，每行显示指标名、判定方式徽章、权重滑条、删除按钮。拖动权重滑条时，该行权重数值即时变化；所有权重之和恒定为 1 的自动归一提示（超出时红字提示并禁用主按钮）。
- 右侧：**加权总分构成预览**。一个环形图实时展示各指标族在总分中的占比，拖滑条时环形图扇区平滑伸缩，让用户直观看到"这次评测偏重什么"。
- 下方：**硬失败门槛**区——列出所有被标为硬失败的规则（如禁用工具调用、安全 Judge 否决），说明"任一命中，整条用例直接判失败"；**接受规则**说明卡：优化环节生效的机械验收标准（训练集提升 ≥ 0.03、回归集零硬失败、无新增失败、时延不超 1.5 倍）在此预告，与第 5 环节呼应。

**交互与动效**：环形图随权重实时动画变化；「保存为模板」成功后有卡片飞入右上角"模板库"图标的动效。

**出口**（随模式变化）：独立建设模式下为「保存评测集」+「导出」（评测集入库并在页签上显示资产角标）；流程引导模式下为「确认指标组合，进入数据集创建」→ 流程栏中环节①打勾 → 进入环节②"数据集创建"首页（页面 2.1）。

## 页面 2.3 自动生成配置页（双生成器）

**页面价值**：这是本环节的核心价值页——回答"针对每条业务规则，造什么用例、为什么造"。让规则与真实业务数据组合，自动倒推出完整用例。

**布局与元素**：

1. **生成器选择区**：两张可选卡（可单选也可都选）：
   - **规则组合生成器**（徽章：deepeval 式代码评估）：基于等价类、边界值、状态迁移、正交、决策表、场景法、因果图 7 种测试方法，把业务规则的因子展开成确定性用例——每条用例的期望结果是可代码判定的；
   - **AI 生成器**（徽章：G-Eval）：由大模型按业务场景语义创造用例（含对抗、边界、易混淆样本），期望结果由 AI 评判标准兜底。
   - 都选时下方出现配比滑条（如"规则组合 60% / AI 生成 40%"）。
2. **规则选择区**：左侧列出环节①确认的全部业务规则（预置 + 自定义），逐条勾选"要为哪些规则造用例"；每条规则旁标注该规则缺的覆盖（如"尚无边界值用例"）。
3. **业务数据样本区**：选择真实业务数据（如贷款申请样例、对话日志样本），支持上传样本文件或从工作区已有数据勾选。规则 × 样本的笛卡尔组合以矩阵预览：行是规则、列是样本，格子显示将生成的用例数。
4. **生成逻辑说明卡**（折叠面板，默认展开一半）：用自然语言解释生成逻辑——"系统会根据你要验证的目标结果，倒推这条用例需要触发哪些工具、期望什么输出：例如验证'高风险申请必须人工复核'，会生成一条高风险样本输入，期望工具链包含风险分类工具，期望输出包含人工复核提示"。配一张"目标结果 → 倒推工具调用链 → 组装用例"的三步示意图。
5. **生成规模与参数**：总用例数、单条用例最大步数、多轮对话用例占比（串行用例会在此声明轮数）。

**交互与动效**：规则×样本矩阵的格子在勾选规则/样本时实时点亮；配比滑条拖动时两生成器卡片面积此消彼长。

**调用的底层能力**：子 skill `requirements-analysis`（UC 15 字段块 + testspec 4 表）、`test-case-generator`（因子提取 + 7 方法库路由 + 五层断言生成）；机械层 `generate_requirements.py`、`generate_testcases.py`。

**出口**：「开始生成」→ 进入页面 2.4 生成过程页。

## 页面 2.4 生成过程与置信度页

**页面价值**：生成过程透明化，让用户对每条用例的可信程度有预期，知道哪些需要重点人工复核。

**布局与元素**：

- 顶部：总进度条 + 当前生成到哪条规则 × 哪个样本的实时字幕（"正在为『风险分类准确性』×『样本-企业贷款-003』生成边界值用例…"）；
- 主体：用例卡片**逐条流入**的瀑布流，每生成一条，卡片从顶部落入列表，卡片上包含：用例名、来源徽章（规则组合 / AI 生成）、命中的业务规则、一句话测试意图、**AI 置信度评分**（0-100%）；
- 置信度视觉分级：≥85% 绿色、60-85% 黄色、<60% 红色并在卡片右上角带"建议重点复核"角标；

- 生成结束后弹出小结条："共生成 N 条（规则组合 X / AI 生成 Y），平均置信度 Z%，W 条建议人工复核"。

**交互与动效**：卡片落列有物理弹跳感；叠放时多张卡片带阴影错位层叠，展开时扇形铺开；置信度数字滚动计数。

**出口**：「进入人工审核」→ 页面 2.5。

## 页面 2.5 用例审核与人工标注页

**页面价值**：AI 造的数据必须经过人——本页完成用例的逐条确认、修改与 GT 标注，是数据集质量的最后闸门。

**布局与元素**：

- 左侧：**用例清单**（含导入 + 生成的全部用例），支持按来源 / 置信度 / 业务规则 / 质量分筛选；每条用例一个卡片，卡片左上角是**12 维质量评级的红黄绿角标**（与测试执行页签的用例自检角标同一视觉语言），评级来自自动质检（字段完整性、步骤数量、验证性词汇等 9 个标准维 + 工具覆盖率、工作流覆盖率、记忆覆盖率 3 个 Agent 专属维）。

- 低质量用例处理：质量分低于阈值的用例集中在清单顶部"待处理"区，支持「重新生成」（触发底层重生成）、「手动修复」、「废弃」三个快捷操作。

**调用的底层能力**：`case_quality_checker.py`（12 维质量评分）、标注流程（`annotate_server.py` / `annotator.py` 能力页内化）、`case_io.py`（cases YAML 读写）。

**出口**：「确认数据集」→ 页面 2.6。

# 第三部分：评估 Agent 执行

> 本环节回答"**怎么评**"。分两个子环节：先**适配自检**（确认待测系统能接到多深），再**批量执行 + 诊断**（跑完整个数据集并归因）。

## 页面 3.1 待测系统适配页（三层级适配）

**页面价值**：评测的深度取决于接入的深度。本页让用户看清三个适配层级分别能拿到什么数据、支撑什么指标，并选择/配置自己能接的层级。

**布局与元素**：

- 页面主体是**三张层级卡，自上而下观测能力递进**，视觉上像三层楼，越往下颜色越深、点亮的指标越多：
  1. **黑盒（对话接口 / 网页）**：只通过对话接口或网页与待测 Agent 交互，拿到输入和最终回答。配置项：接口地址 / 网页入口、请求模板、超时时间。卡下方列出该层级可支撑的指标（任务成功、答案相关性、输出结构合规等业务结果类指标）。
  2. **半白盒（日志转换为标准格式）**：把待测系统的运行日志转换为标准 trace 格式（UATR / NDJSON），能看到工具调用序列、每步耗时、模型调用。配置项：日志来源路径 / 拉取方式、字段映射模板（提供"日志字段 → 标准 trace 字段"的映射表，可从样本日志自动识别预填）。可支撑的指标追加：工具调用正确性、步骤效率、工具效率、Token 消耗等过程类指标。
  3. **白盒（功能插桩）**：在待测 Agent 内部插桩（prompt 渲染、skill 触发、memory 读写、advisor 链全观测）。配置项：插桩方式（内置适配器直连 OpenLab Robot 子进程 / Spring AI HTTP / mock）、插桩点勾选。可支撑的指标追加：全部指标 + F1-F8 完整失败归因 + 9 Judge 全量评审 + 白盒档用例。
- 每张卡右侧有一个"指标点亮条"：直观显示选这一层后环节①评测集中有多少指标会被激活、多少会灰掉（灰掉的标注"需更深适配层级"）。
- 底部说明条："三层级可以同时配置，系统按'能深则深'逐条用例取最优可用层级；黑白灰三档用例（环节②标记）会自动匹配对应层级。"

**交互与动效**：层级卡展开/收起手风琴动效；配置完成一层，该层卡片边缘亮起、指标点亮条逐个点亮；日志字段映射有拖拽连线动效。

**调用的底层能力**：`adapters/`（openlab_robot / spring_ai_http / mobile_bank_http / mock）、trace 合约（guide 02）。

**出口**：「运行适配自检」→ 页面 3.2。

## 页面 3.2 适配自检报告页（冒烟用例）

**页面价值**：正式跑几百条用例之前，先跑**一条简单冒烟用例**验证三个层级的适配是否真的通，避免大批量执行后才发现数据拿不全。

**布局与元素**：

- 顶部：自动选取的一条冒烟用例卡（默认取数据集中步骤最少的一条，可换）；
- 中部：执行过程实时回显（该用例在待测系统上的实际执行过程，类似测试执行页签的单用例执行态，带蓝色脉冲动效与假进度条）；
- 执行完成后展示**三层级适配检查结果表**，每行一个层级：
  - 状态列：✅ 通过 / ⚠️ 降级可用 / ❌ 不可用；
  - 数据列：该层级实际捕获到的数据摘要（黑盒：最终回答 1 条；半白盒：trace 事件 X 条、识别工具调用 Y 次；白盒：插桩事件 Z 类）；
  - 问题列：未通过时给出具体原因（如"日志中缺少 tool_result 字段，请检查字段映射第 3 行"）和「返回调整」快捷按钮；
- 底部：**适配结论卡**——"本次评测将以【半白盒】为主层级执行，评测集 12 项指标中 10 项可用，2 项（Token 消耗、插桩类 Judge）将跳过"，让用户对结果深度有明确预期。

**交互与动效**：三层级逐行检测、逐行亮灯（绿/黄/红）；降级层级黄色呼吸提醒。

**调用的底层能力**：`eval_runner.py`（单用例冒烟执行）、`excel_to_uatr.py`（日志桥接为标准 trace 的试转）。

**出口**：「确认适配，进入批量运行设置」→ 页面 3.3；冒烟未通过时必须「返回调整」，不允许跳过。

## 页面 3.3 批量运行设置页

**页面价值**：复用批量测试执行的成熟交互，把数据集编排成可执行的任务流。本页与《测试执行任务PRD.md》第 2 页"执行任务设置"同一套语言。

**布局与元素**：

- **执行策略**：批次 / 并行设置（批次内全部执行完才进入下一批次，同一批次内可并行，配说明气泡）；重复执行次数、失败重试次数、单批执行个数三个数字设置（与测试执行页签一致）；
- **数据切分**：评测特有的训练集 / 回归集（train / regression）切分比例滑条与抽样方式（支持分层抽样：按业务规则、优先级、黑白灰档分层），并说明用途——train 用于优化迭代，regression 用于验收防回退；
- **任务编排区**：自动生成一版任务列表。每个串行任务独占一行；重复执行、失败重试、有依赖的用例如纸牌叠放（可展开，子任务缩进 + 虚线）；并行任务在同一行；不同批次之间有分割线。允许拖拽调整：拖到任务右侧设为并行，拖到中间/左侧设为分组叠放，拖到分割线间换批次；
- **评测集与数据集回显条**：顶部固定显示"评测集 v1（12 指标）× 数据集 v1.0（N 条用例）"，明确本次跑的是什么组合。

**交互与动效**：拖拽卡片磁吸、叠放扇形展开、批次裂变等动效与测试执行页签保持一致。

**出口**：「开始批量执行」→ 页面 3.4。

# 第五部分：优化（双闭环）

> 本环节回答"**怎么变得更好**"。分两个正交方向：**用例 / 指标优化**（改测试本身，让评测更准）与 **Agent 优化**（改被测系统，让 Agent 更强），并以迭代轮次为主线可视化每一轮的变化。

## 页面 5.1 优化总览页（双闭环入口）

**页面价值**：先讲清优化哲学——评测不好可能是 Agent 不行，也可能是考卷出得不好；两条路都要能走，且互不干扰。

## 页面 5.2 用例 / 指标优化页（用例自优化）

**页面价值**：业界只有 prompt 自优化，没有产品化的用例自优化——本页把这个差异化能力讲透：一轮评测后自动迭代数据集本身。

**布局与元素**：

1. **数据集体检区**：12 维质量分雷达图（9 个标准维 + 工具覆盖率、工作流覆盖率、记忆覆盖率 3 个 Agent 专属维），低分维度红色高亮并可点击展开该维度的问题用例列表；
2. **错误分布与缺口区**：本轮 F1-F8 错误分布图 + **spec 缺口清单**（"业务规则『大额转账二次确认』没有任何边界值用例覆盖"），每个缺口带「生成补漏用例」按钮；
3. **变异杀伤矩阵区（mutation kill matrix）**：系统对 Agent 注入 6 类变异（跳过工具 / 重复工具 / 错误参数 / 空结果 / 幻觉 / 冗余），检验现有用例能不能"杀死"这些变异。矩阵行是用例、列是变异，格子绿 = 该用例能检出此变异、灰 = 检不出。列级检出率低的变异类整列标黄——"你的考卷测不出『幻觉』类问题"；
4. **优化建议清单区**：自动生成三类建议卡——**增强**（某用例补充对抗样本）、**修改**（某用例 GT 过宽，建议收紧断言）、**废弃**（重复或长期无区分度的用例），每条建议带理由与影响面；用户逐条「采纳 / 忽略」，或一键采纳全部；
5. **指标优化子区**：若问题出在指标本身（如评分标准过松导致普遍高分），建议修改指标定义（权重、阈值、G-Eval 评分标准措辞），修改后回写环节①的评测集并生成新版本。

**交互与动效**：雷达图与问题列表联动；杀伤矩阵逐列扫光；采纳建议时建议卡飞入"已采纳"区的动效；应用后弹出"数据集 v1.0 → v1.1"版本升级动画。

**调用的底层能力**：子 skill `test-case-self-optimization`；`case_quality_checker.py`、`case_optimizer.py`（dry-run / apply）、`mutation_generator.py`、`case_io.py`。

**出口**：「生成迭代报告并重跑评测」→ 页面 5.4；或切右门。

## 页面 5.3 Agent 优化页

**页面价值**：把第 3 环节采纳的修改建议变成真实的 patch，并用机械规则验证"改了确实更好、且没改坏"。

**布局与元素**：

1. **优化目标选择区**：五张目标卡（对应"绝不只改 prompt"原则）——**skill**（SKILL.md 描述与触发）、**模型参数**（温度、上下文等）、**调用过程**（workflow / advisor 链）、**工具选择**（tool schema / tool policy）、**记忆**（memory / reference 注入）。待采纳建议已按五类自动归位到各卡下，每卡显示"待改 X 处"。
2. **patch 预览区**：点击任一处改动，展示 diff 预览（改前 / 改后对照），patch 遵循"越小越好"原则；用户可逐处确认，也可全部交给自动优化。
3. **优化执行区**：两种模式（沿用 3.7 的开关）——
   - **自动模式**：系统自动生成 patch → 在 train 集重跑 → 自动判断接受规则 → 通过则应用、不通过则回滚，全程实时滚动日志；
   - **交互模式**：每个 patch 应用前弹确认，用户可修改后再跑。
4. **接受规则仪表盘**：五条机械验收标准以五个仪表灯呈现——train 提升 ≥ 0.03、regression 零硬失败、零禁用工具调用、无新增失败、时延 ≤ 1.5 倍。重跑结果回来后五灯逐一点亮（绿过 / 红拒），任一红灯则该 patch 自动回滚并标注原因；安全 Judge 否决为独立红色总闸（一票否决）。
5. **reference 注入提示**：若优化方向是记忆类，提示"已将高价值参考样例自动注入 reference，预期缩短执行轮数"（对应 F8 执行冗余的优化）。

**交互与动效**：patch diff 左右分栏滑入；五个仪表灯点亮有检定动效；回滚时 patch 卡碎裂回收动画。

**调用的底层能力**：`auto_patcher.py`（A/B 全自动优化）、`reference_optimizer.py`（reference 注入）、`opik_adapter.py`（HRPO 优化器）、`agents/patch-writer.md` / `agents/gatekeeper.md`、接受规则（guide 06）。

**出口**：优化生效后自动触发一轮重测 → 页面 5.4。

## 页面 5.4 迭代轮次可视化页

**页面价值**：让"变好"这件事可被看见——每一轮改了什么、分数怎么动、质量怎么涨，串成一条进化时间线。

**布局与元素**：

- **迭代时间线**（主轴）：横向时间线，每轮一个节点（v1.0 基线 → v1.1 → v1.2 …），节点上标注该轮的闭环类型（用例优化 / Agent 优化）；
- **双曲线图**：加权总分曲线 + 数据集质量分曲线随轮次的变化，两条线双轴呈现，直观看出"考卷变准"与"Agent 变强"各自的贡献；
- **每轮变化卡**：点击任一轮节点，展开该轮的变化明细——采纳的 patch 列表、用例增/改/废弃计数、变异检出率前后对比、各指标分数前后对比（↑↓ 箭头）、五灯验收结果回放；
- **patch 影响矩阵**：行是各轮 patch、列是指标，格子显示该 patch 对指标的增益/回退，一眼看出哪次改动收益最大、哪次有副作用；
- **多轮对比模式**：勾选任意两轮，生成并排对比视图（指标雷达叠加、失败用例集合的差异——修了哪些、新出现哪些）。

**交互与动效**：时间线随轮次增加向右生长；双曲线逐轮描线动画；切换轮次节点时变化卡翻转更新。

**调用的底层能力**：`case_iteration_report.py`（迭代报告）、`dashboard.py`（多轮趋势）、guide 08 图表集（迭代曲线、patch 影响矩阵）。

## 页面 5.5 收尾页（验收 · 沉淀 · 下一轮）

**页面价值**：给本任务一个干净的句号，并铺好持续评测的路。

**布局与元素**：

- **验收小结卡**：当前轮次、最终加权总分与通过率、相对基线的提升、仍开放的问题清单（未解决的失败 + 未采纳的建议）；
- **沉淀区**：本轮最终产物一键归档（评测集版本、数据集版本、patch 记录、迭代报告、完整报告包），产物全部落工作区、Git 可追踪可回滚；
- **CI 回归接入卡**：「纳入持续回归」——把当前评测集 + 数据集固化为回归基线，后续被测 Agent 每次变更可自动重跑回归集并对比基线，防回退；
- **结束动作**：「结束任务」（任务归档，可在左侧一级菜单「执行历史」页签中查看与回看）或「开启新一轮优化」（回到页面 5.1，数据集与评测集沿用最新版本）。

**调用的底层能力**：`ci_regression.py`、`report_manager.py`（归档 CRUD）。

**出口**：任务结束，左侧流程栏五环节全部打勾，播放入库动效；流程栏收起，任务以"已完成"徽标沉淀到「执行历史」页签。

## 设计原则备忘

1. **页面只做编排，能力全在子 skill**：网页是 `skills/agent-eval-v1.1` 的可视化外壳，指标、生成、诊断、优化均调用底层子 skill 与脚本；
2. **以点带面**：五个环节各自是可独立演示的价值点，串起来是完整解决方案，不发散；
3. **证据链贯穿**：任何结论可下钻到用例 ID + trace 事件；
4. **绝不只改 prompt**：优化目标六分（prompt / tool schema / tool policy / workflow / memory / skill）+ 用例本身；
5. **交互语言统一**：卡片叠放、红黄绿角标、流程环节状态、批次裂变等动效与「批量测试执行」页签保持一致，降低学习成本。
6. **独立建设与流程引导并存**：五个大环节默认可从左侧一级菜单单独进入、单独导出产物；点「开始」才唤起流程栏进入顺序引导，流程栏可收起、可从「执行历史」页签恢复。
